# 可观测性：日志、指标、链路追踪

> 可观测性不是“多打日志”，而是让系统出问题时能回答：谁受影响、慢在哪里、为什么慢、怎么止血。

## 一、三大支柱

| 类型 | 回答什么问题 | 典型内容 |
| --- | --- | --- |
| 日志 Logs | 发生了什么 | 错误、关键状态、请求上下文 |
| 指标 Metrics | 系统现在健康吗 | QPS、错误率、延迟、资源使用 |
| 链路追踪 Tracing | 一次请求慢在哪里 | 调用链、span、依赖耗时 |

三者关系：

```mermaid
flowchart LR
    Metrics["指标发现异常"] --> Trace["链路定位慢点"]
    Trace --> Logs["日志确认细节"]
    Logs --> Fix["止血和修复"]
```

## 二、核心指标

服务维度：

- QPS / TPS。
- 错误率。
- P50 / P95 / P99 延迟。
- 超时率。
- 饱和度：CPU、内存、磁盘、连接池、队列。

依赖维度：

- DB 慢查询和连接池等待。
- Redis 命中率、慢命令、大 key、热 key。
- MQ 堆积、消费延迟、失败重试。
- 下游 RPC P99 和错误率。

资源维度：

- CPU user/system/iowait。
- 内存 RSS、heap、OOM。
- 磁盘 await/util。
- 网络重传、RTT、连接状态。

## 三、为什么必须看 P99

平均值会掩盖尾延迟。

```text
99 个请求 10ms
1 个请求 3000ms
平均约 40ms
但用户可能已经明显感知卡顿
```

面试里要强调：

- 在线服务看 P95/P99。
- 批处理看吞吐和完成时间。
- 核心链路看错误率和超时率。
- 队列系统看堆积和消费延迟。

## 四、日志怎么打

好的日志包含：

- trace_id / request_id。
- user_id / order_id 等关键业务标识。
- 操作类型。
- 错误码和错误原因。
- 依赖耗时。
- 关键状态流转。

不要这样：

```text
log.Println("error")
```

要能支持排查：

```text
order_pay_failed trace_id=9f8a7c6b order_id=123 user_id=456 err=timeout cost=1200ms downstream=payment
```

注意：

- 不打印密码、token、身份证、银行卡等敏感信息。
- 高频日志要采样，避免日志把磁盘打满。
- 错误日志要可聚合，不要完全无结构。

## 五、链路追踪

一次请求应该能看到：

```text
HTTP 入口
  -> 参数校验
  -> DB 查询
  -> Redis 查询
  -> RPC 调用
  -> MQ 投递
  -> 响应返回
```

每个 span 关注：

- 开始时间。
- 耗时。
- 状态码。
- 错误信息。
- 下游地址。

链路追踪最适合定位：

- 慢在哪个依赖。
- 是否串行调用太多。
- 是否跨机房调用。
- 是否连接池等待。

## 六、告警设计

告警要关注用户影响，而不是只盯单点资源。

优先级：

1. 错误率升高。
2. P99 延迟升高。
3. 核心业务量异常下降。
4. MQ 堆积和消费延迟。
5. 资源饱和度接近上限。

好的告警：

- 有明确负责人。
- 有影响范围。
- 有排查入口。
- 有处理手册。
- 有抑制和聚合，避免告警风暴。

## 七、事故复盘模板

```text
1. 背景
2. 时间线
3. 影响范围
4. 直接原因
5. 根因
6. 止血动作
7. 长期修复
8. 防复发机制
9. 监控和告警补充
```

复盘不要只写“加强测试”，要落到：

- 哪个指标缺失。
- 哪个保护机制缺失。
- 哪个发布流程缺失。
- 哪个容量评估缺失。
- 谁负责、什么时候完成。

## 八、常见坑

- 只有日志，没有指标和 trace。
- 日志没有 trace_id，无法串起请求。
- 只看平均延迟，不看 P99。
- 告警太多，真正事故被淹没。
- 告警只报机器资源，不报业务影响。
- 敏感信息进入日志。
- 线上问题只能靠猜，缺少可观测性埋点。

## 九、面试表达

```text
我理解可观测性主要解决线上问题定位效率。
指标负责发现异常，链路追踪负责定位慢在哪一段，日志负责还原具体上下文。
核心指标我会看 QPS、错误率、P95/P99、超时率和资源饱和度，而不是只看平均值。
告警设计上优先关注用户影响，比如错误率、P99、核心业务量和 MQ 堆积，然后再看 CPU、内存、磁盘等资源指标。
```
