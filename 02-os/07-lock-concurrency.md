# 锁与并发

> 锁解决并发安全，但也会带来阻塞、死锁、饥饿和性能瓶颈。

## 一、为什么需要锁

多个线程或 goroutine 同时访问共享数据，可能产生竞态。

锁的目标：

- 保证临界区互斥。
- 保护共享状态。
- 保证可见性和顺序。

## 二、常见锁

| 锁 | 特点 | 适合 |
| --- | --- | --- |
| Mutex | 互斥访问 | 通用临界区 |
| RWLock | 多读单写 | 读多写少 |
| SpinLock | 忙等 | 临界区极短 |
| Semaphore | 控制并发数 | 资源池 |
| CAS | 无锁原子操作 | 计数、状态切换 |

## 三、死锁

死锁四个必要条件：

1. 互斥。
2. 持有并等待。
3. 不可剥夺。
4. 循环等待。

```mermaid
flowchart LR
    A["线程 A<br/>持有锁 1"] --> B["等待锁 2"]
    C["线程 B<br/>持有锁 2"] --> D["等待锁 1"]
    B --> C
    D --> A
```

避免：

- 固定加锁顺序。
- 缩小锁范围。
- 不在锁内调用外部服务。
- 使用超时。
- 减少嵌套锁。

## 四、CAS

CAS：Compare And Swap。

```text
如果当前值 == 预期值
  -> 更新为新值
否则失败重试
```

优点：

- 无锁。
- 适合简单原子更新。

问题：

- ABA。
- 自旋重试消耗 CPU。
- 复杂逻辑难写。

## 五、锁性能问题

常见现象：

- 吞吐下降。
- P99 延迟升高。
- CPU 高但业务处理少。
- goroutine 阻塞。

排查：

- Go mutex profile。
- block profile。
- pprof。
- runtime trace。
- 临界区耗时。

## 六、常见坑

- 锁范围太大。
- 锁内做 IO、RPC、数据库操作。
- 读写锁在写多场景更慢。
- 自旋锁用于长临界区。
- CAS 循环没有退避。
- 忘记 defer unlock 或重复 unlock。

## 七、面试表达

```text
锁用于保护共享状态，但会带来阻塞和上下文切换。
Mutex 适合通用互斥，RWLock 适合读多写少，CAS 适合简单原子状态更新。
死锁通常来自多个线程加锁顺序不一致，解决上要固定加锁顺序、缩小锁范围，避免锁内调用外部服务。
在 Go 服务里我会用 mutex profile 和 block profile 排查锁竞争。
```
