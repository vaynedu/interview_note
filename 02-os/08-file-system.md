# 文件系统

> 文件系统题重点是 inode、Page Cache、fsync、顺序写、随机写和崩溃一致性。

## 一、文件系统做什么

文件系统负责：

- 文件命名。
- 目录结构。
- 元数据管理。
- 块分配。
- 权限。
- 缓存。
- 崩溃恢复。

## 二、inode

inode 保存文件元数据：

- 文件大小。
- 权限。
- 所属用户。
- 时间戳。
- 数据块位置。
- 链接数。

文件名存在目录项中，目录项把文件名映射到 inode。

```mermaid
flowchart LR
    Name["文件名"] --> Dir["目录项"]
    Dir --> Inode["inode"]
    Inode --> Blocks["数据块"]
```

## 三、Page Cache

文件读写通常经过 Page Cache。

读：

```text
read -> Page Cache 命中 -> 返回
read -> 未命中 -> 磁盘读取 -> 放入 Page Cache
```

写：

```text
write -> 写 Page Cache -> 返回
后台刷盘
```

所以 write 成功不代表落盘。

## 四、fsync

`fsync` 要求把文件数据和必要元数据刷到稳定存储。

常见场景：

- 数据库事务日志。
- 文件上传完成。
- 关键配置写入。

代价：

- 触发磁盘 IO。
- 频繁 fsync 会降低吞吐。

## 五、顺序写 vs 随机写

顺序写更快：

- 磁盘寻址成本低。
- SSD 内部写放大更可控。
- Page Cache 和预写日志更友好。

随机写慢：

- 写放大。
- IOPS 压力大。
- 延迟更不稳定。

数据库 WAL、Kafka 日志都利用顺序写提升性能。

## 六、日志型文件系统

Journal 记录元数据或数据变更，用于崩溃恢复。

目标：

- 避免系统崩溃后文件系统元数据损坏。
- 提升恢复速度。

注意：

- 文件系统 journal 不等于应用数据一定安全。
- 应用仍需 fsync 或自己的 WAL。

## 七、高频面试题

### inode 是什么？

inode 是文件元数据结构，保存文件大小、权限、时间、数据块指针等。文件名通过目录项映射到 inode。

### write 成功是否代表落盘？

不一定。write 通常只是写入 Page Cache，后台再刷盘。需要 fsync 才能更强地保证落盘。

### 为什么顺序写快？

顺序写减少随机寻址和写放大，更适合磁盘和 SSD 的写入模型，也更容易被系统优化。

## 八、常见坑

- write 成功就认为数据不会丢。
- 日志写入不 fsync，宕机后丢数据。
- 大文件扫描冲刷 Page Cache。
- 小文件太多导致 inode 或元数据压力。
- 删除文件但进程仍持有 fd，磁盘空间不释放。

## 九、面试表达

```text
文件系统通过目录项把文件名映射到 inode，inode 再指向数据块。
文件读写通常会经过 Page Cache，所以 write 成功通常只是写到内存缓存，不代表已经落盘。
如果要保证关键数据落盘，需要 fsync，但频繁 fsync 会影响性能。
顺序写比随机写快，所以数据库 WAL、Kafka 日志都会利用顺序追加写。
```
