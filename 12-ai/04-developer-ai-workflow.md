# 程序员如何高效使用 AI

> AI 对程序员最有价值的地方不是“替你思考”，而是把明确任务更快推进到可验证结果。

## 一、适合交给 AI 的任务

适合：

- 阅读陌生代码。
- 生成样板代码。
- 写测试。
- 重构局部模块。
- 补文档。
- 查调用链。
- 总结日志和错误。
- 做代码 review。
- 生成迁移方案。

不适合直接放手：

- 需求不清的核心架构决策。
- 金融资金类代码。
- 安全敏感改动。
- 无测试覆盖的大规模重构。
- 权限很高的线上操作。

## 二、任务拆解模板

给 AI 任务时，最好包含：

```text
目标：
范围：
上下文：
约束：
验证方式：
不要做什么：
```

例子：

```text
目标：给订单查询接口增加 merchant_id 过滤。
范围：只改 order service 和对应测试。
约束：不要改数据库 schema。
验证：运行 go test ./internal/order/...
不要做：不要重构无关代码。
```

## 三、开发闭环

```mermaid
flowchart LR
    Req["需求"] --> Plan["让 AI 拆计划"]
    Plan --> Code["小步改代码"]
    Code --> Test["运行测试"]
    Test --> Review["AI / 人工 Review"]
    Review --> Fix["修复"]
    Fix --> Done["提交"]
```

关键：

- 小步提交。
- 每步可验证。
- 不让 AI 一次性改太大。
- 失败时让 AI 基于错误输出修复。

## 四、让 AI 写代码的正确方式

不要说：

```text
帮我实现这个功能。
```

更好：

```text
先阅读相关文件，找出当前模式。
只修改 A/B/C 文件。
保持现有接口不变。
补充单元测试。
运行指定测试。
最后总结改动和风险。
```

## 五、AI Code Review

适合让 AI 检查：

- 空指针。
- 错误处理。
- 并发安全。
- 事务边界。
- 幂等。
- 兼容性。
- 测试缺失。
- 日志和监控。

Review 提示：

```text
请以代码审查方式检查这个 diff。
只输出问题，按严重程度排序。
每个问题给出文件、行号、原因和建议。
重点关注数据一致性、并发、错误处理和测试缺失。
```

## 六、排障场景

给 AI：

- 错误日志。
- 最近变更。
- 监控截图或指标。
- 相关代码。
- 已尝试步骤。

让 AI：

- 列可能原因。
- 给排查顺序。
- 标出最可能根因。
- 给止血方案和长期修复。

## 七、AI 使用红线

- 不提交没看过的代码。
- 不把密钥、生产数据直接发给外部模型。
- 不让 AI 绕过权限审批。
- 不接受未验证的“已修复”。
- 不让 AI 大规模重构无测试系统。
- 不把 AI 输出当作事实来源。

## 八、个人工作流建议

### 1. 日常编码

```text
人：定方向、拆任务、做判断
AI：读代码、生成初稿、补测试、修小错
人：review、压测、上线判断
```

### 2. 学习新项目

```text
让 AI 先画模块图
再追调用链
再总结关键数据模型
最后让它列风险和后续事项
```

### 3. 维护个人 Skills

把反复说的话沉淀：

- 代码风格。
- 测试命令。
- Review 标准。
- 文档模板。
- 排障流程。

## 九、面试表达

```text
我使用 AI 的方式不是让它自由写代码，而是把它放进工程闭环。
我会先明确任务范围、上下文、约束和验证方式，让 AI 小步修改，然后运行测试和 review。
对重复流程，我会沉淀成 AGENTS.md 或 Skill，比如代码评审、排障、生成接口模板。
核心原则是：AI 提升速度，人负责判断、边界和最终质量。
```
