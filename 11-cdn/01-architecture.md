# CDN · 架构与原理

> CDN 是什么 / 核心组件 / 数据流 / 关键指标 / 大厂方案对比

## 〇、核心提炼（5 段式）

### 核心机制（4 条必背）

1. **三层节点结构** - 边缘节点（L1，最近用户）+ 区域节点（L2，省级 / 大区）+ 源站（Origin），按层级回源
2. **智能调度（GSLB）** - 基于 LDNS IP / 用户地理位置 / 运营商 / 实时负载 / 节点健康度，把用户路由到最近最健康的边缘
3. **缓存命中率决定一切** - 命中率 > 90% 是基础，>95% 是优秀，决定了**回源带宽 + 用户延迟 + 成本**
4. **回源优化机制** - 回源合并（多个边缘同时回源同一资源时合并成 1 个）+ 预热（活动前主动推到边缘）+ 防回源风暴

### 核心本质（必懂）

> CDN 的本质是 **"把内容缓存到离用户最近的地方，用就近访问替代回源"**：
>
> - **本质是分布式缓存网络**：全球 N 个边缘节点 + 多层回源 + 智能调度
> - **解决的核心问题**：
>   - **延迟**：用户 → 源站可能跨大洲（200ms+），用户 → 边缘节点是同城（5-20ms）
>   - **带宽成本**：源站只需 1 份输出（被各边缘缓存）vs 直接服务 = 用户量 × 资源大小
>   - **源站保护**：DDoS / 突发流量被边缘吸收
>
> **CAP 视角**：
> - CDN 是 AP 系统（牺牲一致性保可用）
> - 缓存可能存在不一致窗口（边缘节点之间 / 边缘与源站）
> - 强一致内容 → 不要走 CDN（或刷新 / Cache-Control: no-cache）
>
> **关键事实**：
> - **静态内容是 CDN 主场**：图片 / 视频 / JS / CSS，命中率可达 99%
> - **动态内容也能 CDN**：但需要"动态加速"（路由优化 + TCP 优化 + 边缘计算）
> - **CDN 不是万能**：实时性强 / 个性化强的内容（推荐 / 评论流）不适合
> - **国内 CDN 复杂**：跨运营商 / 跨南北 / 跨境

### 完整流程（面试必背）

```
用户访问 CDN 的完整路径:

1. DNS 解析（GSLB 调度）:
   user → 本地 DNS（LDNS）
   LDNS → CDN 的权威 DNS
   权威 DNS 判断:
     - LDNS IP 推断用户位置 + 运营商
     - 查询节点健康状态 / 实时负载
     - 返回最优边缘节点 IP

2. 边缘节点查询:
   user → 边缘节点 IP（建 TCP / TLS）
   边缘节点 GET /resource:
     - 查本地缓存 → 命中 → 直接返回（毫秒级）
     - 未命中 → 走回源流程

3. 回源流程（Cache Miss）:
   边缘 → 区域节点（L2）
   L2 查缓存 → 命中 → 返回 + 缓存到边缘
            → 未命中 → 回源到源站

   源站接收请求:
     - 处理业务
     - 返回响应
     - 响应头中 Cache-Control 决定缓存策略

   响应链路:
   源站 → L2（缓存）→ 边缘（缓存）→ user

4. 后续请求（缓存命中）:
   user → 边缘 → 直接返回（不回源）
   → TTFB（首字节时间）<= 20ms

防回源风暴（多用户同时 miss 同一资源）:
  - 回源合并: 一个边缘节点同一资源同一时刻只回源 1 次
  - 其他请求等待这个回源结果
  - 拿到后一起返回 + 缓存

预热（主动推送，应对突发）:
  - 活动前管理后台触发预热
  - CDN 提前从源站拉取资源到所有边缘
  - 活动开始 = 命中率 100%
```

### 4 条核心机制 - 逐点讲透

#### 1. 三层节点结构（为什么是三层）

```
设计哲学:
  全球节点完全 mesh 互联 = O(N²) 连接，太复杂
  分层 = O(N) 连接，可管理

L1 边缘节点（最近用户）:
  - 数量最多（几百到几千）
  - 服务终端用户
  - 缓存最热资源
  - 部署在 ISP / IDC（CMCC / CT / CU）

L2 区域节点:
  - 数量中等（几十）
  - 服务边缘节点的回源
  - 缓存热资源
  - 跨运营商部署

源站（Origin）:
  - 客户自有（业务方）
  - 服务区域节点回源
  - 真实数据源

为什么不直接边缘 → 源站:
  - 几千边缘同时回源 → 源站打挂
  - 跨城跨网回源 → 延迟高
  - 浪费跨城带宽
  → L2 起聚合 + 加速作用

典型分布（中国）:
  L1: 200+ 城市
  L2: 30+ 省级中心
  源站: 客户自定（北上广深通常）
```

#### 2. 智能调度（GSLB）

```
传统 DNS 局限:
  - 返回固定 IP
  - 不考虑用户位置 / 负载

GSLB（Global Server Load Balancer）:
  返回因用户而异的 IP

判断维度（综合）:
  1. 地理位置:
     LDNS IP → 数据库查地理位置
     就近原则（距离最短）

  2. 运营商:
     电信 / 联通 / 移动用户路由到对应节点
     避免跨网（跨网延迟 + 成本）

  3. 节点健康:
     实时探测节点存活 + 延迟
     摘除不健康节点

  4. 实时负载:
     节点 CPU / 带宽接近上限 → 路由到其他节点

  5. 业务策略:
     灰度发布 / 调试环境特定路由

GSLB 实现方式:
  - DNS 智能解析（最常用）: 返回不同 IP
  - HTTP DNS: 业务直接调 HTTPS API 查 IP（绕过 LDNS 不准问题）
  - Anycast: 同一 IP 全球路由就近（CDN + DDoS 防护）

LDNS 不准的坑:
  用户在北京但用了广州的 LDNS → 解析到广州节点
  → 解法: HTTP DNS 直接传客户端 IP
```

#### 3. 缓存命中率（CDN 的生命）

```
命中率计算:
  命中率 = (命中数 / 总请求数) × 100%
  回源率 = 1 - 命中率

健康水位:
  > 95%: 优秀
  90-95%: 良好
  85-90%: 一般
  < 85%: 必须优化

影响命中率的因素:
  1. 缓存键设计:
     URL 含随机参数（如 timestamp）→ 每次不同 → 命中率为 0
     → 缓存键归一化（去除无关参数）

  2. TTL 设置:
     太短 → 频繁回源
     太长 → 内容更新不及时
     → 静态资源长 TTL（1 天 - 1 年）+ 内容 hash URL（如 main.abc123.js）

  3. 资源大小分布:
     大文件: 命中率高（每个用户都要全量）
     小文件: 易碎片化命中

  4. 突发流量:
     活动期 → 新资源激增 → 命中率骤降
     → 预热 + 回源合并

命中率与业务指标:
  命中率从 90% → 95% = 回源流量减半 → 成本节省 50%
  → CDN 优化的核心
```

#### 4. 回源优化（防风暴）

```
回源风暴问题:
  10w 用户同时访问 1 个未缓存资源
  10w 个边缘节点同时回源源站
  → 源站瞬间被打挂

回源合并（Request Coalescing）:
  边缘节点同一资源:
  - 第 1 个 miss 请求触发回源
  - 后续 miss 请求等待第 1 个完成
  - 拿到结果后所有请求一起返回 + 缓存
  → 源站只承受 1 次回源

预热（Pre-warming）:
  活动前/突发前主动推:
  - 管理后台触发预热 API
  - CDN 系统从源站拉取资源
  - 推送到所有边缘节点
  - 活动开始 = 命中率 100%

  典型场景:
  - 双 11 商品图预热
  - 直播开始前流推到边缘
  - APP 新版本资源预热

防风暴其他措施:
  - 限流: 边缘节点回源 QPS 限制
  - 排队: 超限请求排队 / 拒绝
  - 降级: 源站挂 → 返回旧缓存（即使过期，stale-while-revalidate）

性能指标:
  TTFB（Time To First Byte）: 首字节时间
    - 缓存命中: < 20ms
    - 缓存未命中: 几百 ms（要回源）

  下载完成时间:
    - 看资源大小 + 带宽
```

### 一句话总结

> CDN 的核心是：**三层节点（边缘 L1 + 区域 L2 + 源站）+ 智能调度（GSLB 多维选最优）+ 缓存命中率（决定一切）+ 回源优化（合并 + 预热）**，
> 本质是**分布式缓存网络**：把内容缓存到离用户最近的地方，**用就近访问替代回源**。
> **核心目标**：低延迟（同城 5-20ms vs 跨大洲 200ms+）+ 低成本（源站只输出 1 份）+ 抗 DDoS（边缘吸收）。
> **关键指标**：命中率 > 95% 是优秀，命中率提升 5% = 回源流量减半 = 成本节省 50%。
> **CDN 不是万能**：静态资源主场（命中率 99%），动态内容用动态加速（路由 + TCP 优化 + 边缘计算）。

## 一、CDN 是什么

### 1.1 一句话定义

> **CDN（Content Delivery Network）= 把内容缓存到离用户最近的边缘节点，用就近访问替代回源**

```mermaid
flowchart LR
    subgraph NoCDN[无 CDN]
        U1[北京用户] -.跨地域.-> S1[(广州源站)]
        U2[新疆用户] -.跨地域.-> S1
        U3[海外用户] -.跨洋.-> S1
    end

    subgraph WithCDN[有 CDN]
        U4[北京用户] --> E1[北京边缘]
        U5[新疆用户] --> E2[乌鲁木齐边缘]
        U6[海外用户] --> E3[东京边缘]
        E1 -.miss.-> S2[(广州源站)]
        E2 -.miss.-> S2
        E3 -.miss.-> S2
    end

    style WithCDN fill:#9f9
```

### 1.2 解决什么问题

| 痛点 | CDN 怎么解 |
| --- | --- |
| **跨地域延迟高** | 边缘节点就近响应（RTT 从几百 ms → 几十 ms） |
| **源站带宽爆** | 边缘命中分担流量，源站只回源一份 |
| **跨网慢**（电信→联通） | 多线 BGP 节点 + 智能调度 |
| **DDoS 攻击** | 流量在边缘清洗 |
| **跨境慢/丢** | 海外节点 + 优化协议（QUIC） |
| **抖动/丢包** | 私有网络回源 + 链路优化 |

### 1.3 不解决什么问题

- **动态、个性化**内容（购物车、用户首页）— 需要"动态加速"或不上 CDN
- **写请求**（POST 提交订单）— 直接回源，CDN 只能加速链路
- **强一致**（金融账本）— CDN 是缓存，天然滞后
- **小流量**冷启动（边缘也要回源拉一次，用户少时优势小）

## 二、整体架构

### 2.1 三层结构

```mermaid
flowchart TB
    subgraph User[用户]
        U1[终端]
    end

    subgraph Edge[边缘节点 L1]
        E1[北京]
        E2[上海]
        E3[广州]
    end

    subgraph Parent[父层节点 L2]
        P1[华北父层]
        P2[华东父层]
        P3[华南父层]
    end

    subgraph Origin[源站]
        O[(用户源站)]
    end

    U1 -->|hit| E1
    U1 -->|hit| E2
    E1 -->|miss| P1
    E2 -->|miss| P2
    E3 -->|miss| P3
    P1 -->|miss 回源| O
    P2 -->|miss 回源| O
    P3 -->|miss 回源| O

    style Edge fill:#9f9
    style Parent fill:#9ff
```

| 层 | 角色 | 部署 |
| --- | --- | --- |
| **边缘节点 L1** | 直接服务用户 | 上千节点，覆盖城市/运营商 |
| **父层 L2** | 边缘 miss 后的二级缓存，收敛回源 | 几十个，按大区部署 |
| **源站** | 原始内容 | 用户自建/对象存储 |

**为什么要父层？**
- 边缘节点上千，全部直接回源 → 源站打爆
- 父层把 N 个边缘 miss 收敛成一次回源（**源站盾**）
- 父层命中率 60-80%，再 miss 才到源站

### 2.2 核心组件

```mermaid
flowchart TB
    DNS[GSLB / 智能 DNS<br/>就近调度]
    Cache[缓存系统<br/>命中/失效/分级]
    Origin[回源系统<br/>合并请求/链路优化]
    Cert[证书系统<br/>TLS/SNI]
    Sec[安全系统<br/>WAF/DDoS/盗链]
    Log[日志/计量<br/>实时分析]
    Edge[边缘计算<br/>EdgeWorker]
    Console[控制台/API<br/>配置下发]

    Console --> DNS & Cache & Origin & Cert & Sec & Edge

    style Cache fill:#9f9
    style Origin fill:#9ff
```

每个都是独立子系统，CDN 厂商技术深度都在这里。

### 2.3 一次请求的完整路径

```mermaid
sequenceDiagram
    participant U as 用户
    participant DNS as 智能 DNS
    participant E as 边缘节点
    participant P as 父层节点
    participant O as 源站

    U->>DNS: 查 cdn.example.com
    DNS->>DNS: 看用户 LDNS 位置 + 运营商
    DNS->>U: 返回最近边缘 IP（如 北京电信）

    U->>E: GET /image.jpg
    E->>E: 查本地缓存

    alt 命中
        E->>U: 返回内容（HIT）
    else 未命中
        E->>P: 回源到父层
        P->>P: 查父层缓存
        alt 父层命中
            P->>E: 返回内容
            E->>E: 缓存到本地
            E->>U: 返回内容（MISS+父HIT）
        else 父层也未命中
            P->>O: 回源源站
            O->>P: 返回内容
            P->>P: 缓存
            P->>E: 返回
            E->>E: 缓存
            E->>U: 返回内容（MISS+父MISS）
        end
    end
```

**关键节点**：
- 用户体验差异：HIT < 父HIT < 全 MISS
- 命中率优化目标：**边缘命中率 > 90%**，**回源率 < 10%**

## 三、关键指标

### 3.1 核心指标

```mermaid
mindmap
  root((CDN 指标))
    性能
      首字节时间 TTFB
      下载速度
      命中率
    可用性
      请求成功率
      连接成功率
    流量
      带宽峰值
      流量总量
      回源带宽
    成本
      流量成本
      请求成本
      回源成本
    安全
      攻击拦截率
      误杀率
```

### 3.2 必须监控的 7 个指标

| 指标 | 含义 | 健康值 |
| --- | --- | --- |
| **边缘命中率** | 边缘命中数 / 总请求数 | > 90% |
| **整体命中率** | (边缘+父层) 命中 / 总请求 | > 95% |
| **回源率** | 回源请求 / 总请求 | < 10% |
| **回源成功率** | 成功回源 / 总回源 | > 99.9% |
| **TTFB** | 首字节时间 | < 100 ms |
| **请求成功率** | 2xx+3xx / 总请求 | > 99.9% |
| **带宽峰值** | 容量规划依据 | 不超签约 |

### 3.3 命中率公式

```
边缘命中率 = HIT / (HIT + MISS + EXPIRED)
回源率 = 回源请求数 / 总请求数 = 1 - 整体命中率（近似）
```

### 3.4 命中率为什么重要

- 命中率 90% → 回源 10%，源站轻松
- 命中率 50% → 回源 50%，源站打爆
- 命中率每提升 10%，回源带宽降一半左右（具体看流量分布）

**优化命中率是 CDN 运维的核心 KPI**。

## 四、数据流详解

### 4.1 静态内容数据流

```mermaid
flowchart LR
    Origin[(源站<br/>image.jpg)]
    Origin -->|首次拉取| Parent[父层]
    Parent -->|分发| Edge[边缘]
    Edge -->|命中| User[用户]

    Note["静态内容：一次拉取，多次分发"]
```

适合：图片、CSS、JS、视频、下载文件。

### 4.2 动态内容数据流（动态加速 / DSA）

```mermaid
flowchart LR
    User[用户] --> Edge[边缘]
    Edge -.私网链路.-> Parent[父层]
    Parent -.优化路由.-> Origin[(源站)]

    Note["动态内容不缓存，但走 CDN 私网<br/>用 TCP 优化、链路优选、QUIC 加速"]

    style Edge fill:#9f9
```

适合：API、登录、个性化页面。

**核心**：**TCP 直连源站慢**，CDN 私网用持久连接、智能选路、协议升级把动态请求加速 2-5 倍。

### 4.3 直播流数据流

```mermaid
flowchart LR
    Streamer[主播推流] --> Origin[源站]
    Origin --> P1[父层节点]
    P1 --> E1[边缘 1] --> V1[观众]
    P1 --> E2[边缘 2] --> V2[观众]
    P1 --> E3[边缘 3] --> V3[观众]

    Note["主播 1 路推流，CDN 树状分发到 N 个观众"]
```

低延迟直播 + 边缘转码（HLS / DASH / RTMP / FLV）。

| 类型 | 说明 | 代表 |
| --- | --- | --- |
| **公共 CDN** | 多租户共享 | 阿里云 / 腾讯云 / Cloudflare |
| **专用 CDN** | 单租户独享 | Netflix / Apple 自建 |
| **私有 CDN** | 企业内部使用 | 大厂内网 |
| **混合 CDN** | 公共 + 私有混合 | 视频网站常用 |

## 六、大厂方案对比

### 6.1 公开 CDN 厂商

| 厂商 | 特点 | 强项 |
| --- | --- | --- |
| **Cloudflare** | 全球最大，免费档丰富 | 安全 + Anycast + Workers |
| **Akamai** | 老牌之王，企业级 | 节点最多 + 复杂调度 |
| **Fastly** | 实时边缘配置（< 1s 生效） | API 加速 + VCL 编程 |
| **AWS CloudFront** | 与 AWS 深度集成 | Lambda@Edge + S3 |
| **阿里云 CDN** | 国内龙头 | 全站加速 + 视频生态 |
| **腾讯云 CDN** | 视频强 | 直播 / 点播 + 海外节点 |
| **网宿** | 国内老牌 | 大客户运维 |
| **百度智能云** | 后起 | AI 边缘 |

### 6.2 自建 vs 用云厂商

| | 自建 | 用云厂商 |
| --- | --- | --- |
| 启动成本 | 高（节点+人力） | 低（按量付费） |
| 灵活度 | 完全自主 | 受厂商限制 |
| 成本 | 流量越大越省 | 流量越大越贵 |
| 适合 | 流量超 10Gbps、视频/电商等核心业务 | 中小流量 / 通用场景 |

**典型案例**：
- 字节、腾讯视频、快手、B 站 → 大量自建 + 部分云厂商兜底
- 中小厂 → 全用云厂商
- Netflix → 全自建（Open Connect）

### 6.3 自研 CDN 的核心模块

```
1. 节点采购与建设（机房 / 带宽 / 服务器）
2. 调度系统（GSLB + 智能 DNS）
3. 缓存软件（基于 Nginx / Varnish / 自研）
4. 回源系统（合并 / 限速 / 安全）
5. 监控与配置（实时指标 + 配置下发）
6. 边缘计算平台（V8 / WASM）
```

## 七、典型坑

### 坑 1：以为 CDN 万能

误区：动态接口也用 CDN 加速结果命中率为 0。

**修复**：
- 静态走缓存 CDN
- 动态走动态加速（不缓存，仅链路优化）
- 区分清楚静态 / 动态资源

### 坑 2：缓存键设计糟糕

URL 带 query string 全部缓存 → 命中率极低（每个用户不同）。

**修复**：缓存键去掉无关参数（详见 [02-cache-strategy.md](02-cache-strategy.md)）。

### 坑 3：源站设计没考虑回源

CDN 上线，源站突然被打爆 → 没考虑过回源风暴。

**修复**：
- 父层做请求合并
- 源站加 LB + 限流
- 预热热点内容

### 坑 4：HTTPS 配置错

证书不更新过期 → 全网用户访问失败。

**修复**：自动续签 + 监控。

### 坑 5：埋点 / 计量错误

按命中率算钱 vs 按回源率算钱差几倍 → 财务不一致。

**修复**：明确 SLA + 自己也埋点对账。

### 坑 6：调度策略激进

DNS 把所有用户调到同一节点 → 节点崩。

**修复**：负载均衡考虑节点容量、健康度。

### 坑 7：缓存预热不到位

促销开抢，热点资源没预热 → 短时间高回源 → 雪崩。

**修复**：活动前预热（详见 [08-troubleshooting-cases.md](08-troubleshooting-cases.md)）。

## 八、面试高频题

**Q1：CDN 是什么？解决什么问题？**

把内容缓存到离用户最近的边缘节点，用就近访问替代回源。

解决：跨地域延迟、源站带宽、跨网慢、DDoS、跨境差。

**Q2：CDN 三层架构是什么？**

边缘 L1（直接服务用户）→ 父层 L2（收敛回源）→ 源站。

父层的作用：减少回源次数，做"源站盾"。

**Q3：CDN 命中率怎么算？多少算正常？**

边缘命中率 = HIT / (HIT + MISS + EXPIRED)

健康值 > 90%。整体命中率（含父层） > 95%。

**Q4：什么内容不适合上 CDN？**

- 动态/个性化内容
- 写请求
- 强一致内容
- 频繁变更（命中率上不去）

动态可以用"动态加速"（仅优化链路，不缓存）。

**Q5：CDN 和反向代理的区别？**

| | 反向代理 | CDN |
| --- | --- | --- |
| 部署 | 单个数据中心 | 全国/全球边缘 |
| 距离 | 离源近 | 离用户近 |
| 调度 | 静态配置 | 智能 DNS + GSLB |
| 规模 | 几台-几十台 | 上千节点 |

CDN 本质是**全球化 + 智能调度的反向代理网络**。

**Q6：CDN 怎么把用户路由到最近节点？**

智能 DNS（GSLB）：
- 看用户 LDNS IP 推断地理位置 + 运营商
- 选最近、负载最低的节点 IP 返回
- 详见 [03-routing-dispatch.md](03-routing-dispatch.md)

**Q7：动态加速的原理？**

不缓存，但走 CDN 私网：
- TCP 持久连接（避免握手）
- 智能选路（避开拥塞）
- 协议升级（HTTP/2 / QUIC）
- 压缩 / 头部优化

加速比 2-5 倍。

**Q8：CDN 关键指标有哪些？**

7 个：边缘命中率、整体命中率、回源率、回源成功率、TTFB、请求成功率、带宽峰值。

**Q9：自建 CDN vs 用云厂商怎么选？**

| | 自建 | 云厂商 |
| --- | --- | --- |
| 适合 | 大流量核心业务 | 中小流量 |
| 成本 | 流量越大越省 | 流量越大越贵 |
| 阈值 | 10 Gbps+ | 看业务 |

**Q10：CDN 怎么处理 HTTPS？**

- 用户证书托管在 CDN
- 用户 → 边缘节点 TLS（边缘存证书）
- 边缘 → 源站可以走 HTTP 或 HTTPS
- SNI 用于多证书共享 IP

## 九、面试加分点

- 强调 CDN 是 **缓存 + 调度 + 网络优化** 三位一体
- 三层架构：**边缘 / 父层 / 源站**，父层是源站盾
- 命中率优化是 **CDN 运维核心 KPI**
- 静态走缓存，**动态走动态加速**（仅链路）
- 大厂自建 CDN（Netflix Open Connect / 字节自研）+ 云厂商兜底
- 自研核心：**调度 + 缓存软件 + 回源 + 边缘计算**
- 关键指标 7 个，TTFB / 命中率 / 回源率最重要
- 跨网用 BGP / 多线 / Anycast 解决
- 新协议红利：**HTTP/2 → HTTP/3 QUIC**（详见 [04-protocol-optimization.md](04-protocol-optimization.md)）
- 永远要测**回源风暴** + **预热**（详见 [08-troubleshooting-cases.md](08-troubleshooting-cases.md)）
