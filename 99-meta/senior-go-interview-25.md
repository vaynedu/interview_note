## 1. Go 调度器 GMP 模型怎么理解？

### 标准答案
Go 调度器核心是 **GMP 模型**：

- **G**：goroutine，用户态协程
- **M**：machine，内核线程
- **P**：processor，逻辑处理器，代表运行 Go 代码所需的资源上下文

执行关系可以理解为：

- G 是待执行任务
- M 是执行任务的线程
- P 负责把可运行的 G 交给 M 执行

一个 M 想执行 Go 代码，必须绑定一个 P。
P 的数量通常由 `GOMAXPROCS` 决定。

### 为什么需要 P？
如果只有 G 和 M，那么所有 goroutine 调度都得直接挂在线程上，会导致调度、队列管理、锁竞争都比较重。
P 的引入本质是为了：

- 持有本地可运行队列
- 减少全局锁竞争
- 提高调度局部性
- 支持 work stealing

### 核心机制
- **本地队列**：每个 P 有自己的 runnable queue
- **全局队列**：当本地队列空了，会去全局队列拿任务
- **work stealing**：P 空闲时去偷其他 P 的任务
- **syscall 切换**：goroutine 做系统调用时，M 可能阻塞，P 会被剥离给其他 M 继续执行
- **网络轮询器**：处理 epoll/kqueue/iocp 等 IO 就绪事件

### 高级理解
Go 调度器本质上是在用户态实现一套高效的任务调度系统，其目标不是绝对公平，而是：

- 高吞吐
- 低切换成本
- 尽量减少线程阻塞影响
- 尽量让大量 goroutine 以较小代价并发运行

### 易错点
- 不是一个 goroutine 对应一个线程
- P 不是 CPU 核心本身，而是调度上下文
- `GOMAXPROCS` 控制的是可同时执行 Go 代码的 P 数量，不是 goroutine 数量

### 追问点
- work stealing 怎么偷？
- syscall 阻塞时发生什么？
- netpoller 和调度器怎么配合？
- 为什么 Go 能起几十万 goroutine？

### 背诵版
> Go 调度器采用 GMP 模型，G 是 goroutine，M 是线程，P 是调度上下文。P 持有本地队列并调度 G 给 M 执行，通过本地队列、全局队列、work stealing 和 netpoller，实现大量 goroutine 的高效调度。

## 2. goroutine 为什么轻量？

### 标准答案
goroutine 轻量主要体现在以下几个方面：

#### 1）初始栈很小
线程通常有较大的固定栈空间，而 goroutine 初始栈很小，并且支持动态扩缩容。

#### 2）创建成本低
goroutine 是 runtime 管理的用户态对象，不像线程那样需要频繁陷入内核创建。

#### 3）切换成本低
goroutine 切换主要由用户态调度器完成，不完全依赖内核线程上下文切换。

#### 4）多 goroutine 复用少量线程
通过 GMP 调度模型，大量 goroutine 可以映射到较少线程上执行。

### 本质原因
goroutine 的轻量不是因为"没有成本"，而是因为 Go runtime 把很多调度和资源管理工作搬到了用户态，降低了单位并发任务的系统成本。

### 易错点
- goroutine 不是免费
- 大量 goroutine 仍然会占用内存、调度资源、GC 成本
- 阻塞不当仍会导致泄漏和性能问题

### 追问点
- goroutine 栈怎么增长？
- 多到多少会出问题？
- goroutine 泄漏如何排查？

### 背诵版
> goroutine 轻量主要因为它初始栈小、由用户态调度器管理、切换成本低，而且大量 goroutine 可以复用少量线程执行。

## 3. Go 调度器的抢占是怎么做的？

### 标准答案
Go 早期的调度抢占更偏**协作式**，即 goroutine 需要在函数调用、channel 操作、系统调用等安全点上，runtime 才有机会调度它。

后来 Go 引入了更完善的**异步抢占机制**，避免某些长时间运行、不主动让出的 goroutine 独占 CPU。

### 目的
防止这种情况：
- 某 goroutine 死循环
- 没有函数调用
- 一直不进入调度点
- 导致其他 goroutine 饿死

### 现代理解
Go 现在支持更强的抢占能力，但仍然不是任意指令点都随意打断，而是在相对安全的位置执行抢占。

### 工程意义
抢占机制提升了：
- 调度公平性
- 延迟可控性
- 对 CPU 密集型任务的鲁棒性

### 易错点
- 不要把 Go 抢占理解成完全等价于 OS 线程抢占
- 某些极端代码仍可能表现出调度不均

### 追问点
- 抢占对 GC 有什么帮助？
- 安全点是什么？
- 为什么早期 Go 会出现 goroutine 饿死？

### 背诵版
> Go 早期更偏协作式调度，后来引入异步抢占机制，避免长时间运行的 goroutine 独占 CPU，从而提升调度公平性和系统响应性。

## 4. Go 的 GC 是什么模型？

### 标准答案
Go 的 GC 核心是 **并发三色标记清扫**。

### 三色标记
- **白色**：尚未访问，理论上可能被回收
- **灰色**：已经访问，但其引用对象还没扫描完
- **黑色**：已经访问，且其引用对象都扫描完

### 回收流程概括
1. 从根对象出发开始标记
2. 并发扫描对象引用关系
3. 标记存活对象
4. 清扫不可达对象
5. 回收内存

### 目标
Go GC 的目标不是"绝对零停顿"，而是：
- 尽量缩短 STW
- 在吞吐和延迟之间做工程平衡
- 尽可能和业务并发执行

### 为什么需要写屏障？
因为 GC 标记过程中，用户程序还在修改对象引用关系。
如果不加约束，可能破坏三色不变式，造成对象误回收。

写屏障就是为了在并发标记期间维护对象可达性信息。

### 易错点
- Go 不是传统 stop-the-world GC
- Go 仍然有 STW，只是尽量缩短
- GC 优化的核心常常不是"调 GC 参数"，而是减少分配和逃逸

### 追问点
- 三色不变式是什么？
- 混合写屏障是什么？
- Go 为什么不用分代 GC？
- GC 触发时机是什么？

### 背诵版
> Go GC 采用并发三色标记清扫，通过写屏障维护并发标记过程中的正确性，目标是在保证吞吐的同时尽量缩短停顿时间。

## 5. Go GC 的 STW 在哪几个阶段？

### 标准答案
Go 虽然是并发 GC，但并不意味着完全没有 STW。
通常在一些关键阶段仍需要短暂 STW，例如：

- GC 启动时需要短暂停顿，扫描 root 的准备工作
- 标记终止阶段需要 STW，确保并发标记收敛
- 某些元数据切换和状态切换时需要短暂停顿

### 为什么一定还有 STW？
因为某些瞬间需要获得一个一致性视图，或者完成全局状态切换。
完全无停顿的代价通常非常高，工程上不划算。

### 工程视角
高级面试里重点不是死记阶段名，而是理解：

> Go 的 GC 优化方向是把大部分工作并发化，但在必须全局一致的关键点保留很短的 STW。

### 易错点
- 不能回答"Go 没有 STW"
- 也不能回答"Go GC 还是传统大停顿"
- 正确说法是"有 STW，但被压缩得比较短"

### 追问点
- 什么会导致 STW 变长？
- root 太多有什么影响？
- goroutine 太多会影响 GC 吗？

### 背诵版
> Go GC 不是没有 STW，而是把 STW 压缩到少数关键阶段，比如启动和标记终止时，用于完成全局状态切换和一致性保证。

## 6. Go 为什么没有传统分代 GC？

### 标准答案
这是一个很经典的高级问题。
传统分代 GC 的假设是：**大多数对象朝生夕死**，所以把年轻代和老年代分开回收会更高效。

Go 没有采用典型的分代 GC，主要是因为它更关注：

- 实现复杂度
- 写屏障成本
- 并发回收的整体收益
- 与 goroutine/栈管理的配合

### 为什么 Go 没走典型分代路线？
可以从工程角度回答：

#### 1）分代 GC 会引入更复杂的屏障和维护成本
尤其在高并发场景，维护代际关系和 remembered set 会比较重。

#### 2）Go 更重视低停顿和工程简单性
而不是做一个非常复杂的 GC 体系。

#### 3）Go 的对象模型和业务场景与 JVM 有差异
Go 服务端很多性能问题可以通过减少分配、对象复用、逃逸优化来解决，不一定非要靠复杂的分代模型。

### 但注意
这不是说 Go GC 不先进，而是说它选择了不同的工程取舍。

### 背诵版
> Go 没有采用典型分代 GC，核心原因是分代机制会增加实现复杂度和屏障成本。Go 更强调并发标记清扫、低停顿和整体工程权衡。

## 7. 什么是写屏障？为什么 GC 需要写屏障？

### 标准答案
写屏障可以理解为：
在程序修改对象引用关系时，插入一小段额外逻辑，让 GC 能感知这些变化。

### 为什么需要？
因为 Go 的 GC 是并发标记，GC 在扫描对象图的同时，用户程序还在继续运行、继续修改指针引用。

如果没有写屏障，可能会发生：

- 某对象本来还可达
- 但在 GC 扫描过程中引用关系被改掉
- 导致 GC 误判为不可达并错误回收

### 本质作用
写屏障用于维护三色标记过程中的正确性，避免"黑对象指向白对象但未记录"的情况破坏三色不变式。

### 易错点
- 写屏障不是"加锁"
- 它是 GC 正确性的辅助机制，不是并发同步原语

### 追问点
- 插入写屏障和删除写屏障区别？
- Go 的混合写屏障是什么？
- 写屏障对性能有什么影响？

### 背诵版
> 写屏障是在对象引用变更时插入的一段辅助逻辑，用来保证并发 GC 标记过程中的正确性，避免对象被误回收。

## 8. Go 内存分配是怎样的？

### 标准答案
Go 的内存分配体系分层比较明显，runtime 会根据对象大小和用途采用不同策略。

### 大致理解
- 小对象优先从本地缓存分配，减少锁竞争
- 不同大小类别有不同 span 管理
- 大对象走更特殊的分配路径
- 最终底层向操作系统申请页内存

### 典型结构
可以从这些层面理解：
- **mcache**：线程/执行上下文本地缓存
- **mcentral**：中心缓存
- **mheap**：全局堆

### 分配目标
- 尽量减少全局锁竞争
- 提高小对象分配效率
- 配合 GC 和 span 管理提高整体内存利用率

### 高级面试答法
> Go 小对象分配会优先走本地缓存，减少锁开销；不足时再向中心缓存和全局堆申请。整体设计和 tcmalloc 思路比较接近，目标是平衡分配效率、并发性能和 GC 管理。

### 追问点
- 什么是 span？
- tiny allocator 是什么？
- 大对象分配路径有什么不同？

### 背诵版
> Go 的内存分配采用分层缓存结构，小对象优先从本地缓存申请，不够再走中心缓存和全局堆，以减少锁竞争并提升分配效率。

## 9. 逃逸分析在性能优化里为什么重要？

### 标准答案
逃逸分析决定变量是分配在栈上还是堆上。
如果对象逃逸到堆，就会增加：

- 堆分配开销
- GC 扫描和回收压力
- 内存局部性损失

### 为什么关键？
因为 Go 服务端很多性能问题，最后不一定是 CPU 算得慢，而是：
- 分配太多
- GC 太频繁
- 堆对象太多

所以性能优化里，逃逸分析是一个非常重要的切入点。

### 常见逃逸场景
- 返回局部变量指针
- interface 装箱
- 闭包捕获变量
- 传入未知生命周期对象
- 反射相关操作

### 怎么看？
```bash
go build -gcflags="-m"
```

### 背诵版
> 逃逸分析决定对象分配在栈还是堆。堆对象越多，GC 压力越大，所以性能优化时经常要关注哪些对象发生了逃逸以及能否减少逃逸。

## 10. Go 的内存模型你怎么理解？

### 标准答案
Go 内存模型核心回答的是：

> 在并发程序里，一个 goroutine 对变量的写，什么时候对另一个 goroutine 可见？

### 核心思想
如果没有同步事件，多个 goroutine 之间对共享变量的读写顺序和可见性就不能被保证。

### 典型同步原语
这些操作会建立 happens-before 关系：
- channel send / receive
- mutex unlock / lock
- `sync.Once`
- `WaitGroup` 中的同步语义
- atomic 操作
- goroutine 启动的一些顺序保证

### 面试要点
内存模型不是在问"变量放哪"，而是在问：
- 可见性
- 有序性
- 同步边界

### 高频总结
> 如果没有同步，就不要假设一个 goroutine 对共享变量的写能立即被另一个 goroutine 看到。

### 易错点
- `time.Sleep` 不是同步原语
- "我测试没问题"不代表符合内存模型
- 数据竞争下程序行为未定义式不可靠

### 背诵版
> Go 内存模型描述的是并发读写下的可见性和有序性。只有通过锁、channel、atomic 等同步手段建立 happens-before，才能保证一个 goroutine 的写对另一个 goroutine 可见。

## 11. happens-before 怎么理解？

### 标准答案
happens-before 是并发程序里一个非常重要的概念，表示：

> 如果 A happens-before B，那么 A 的结果对 B 可见，并且 A 的执行顺序先于 B。

### 在 Go 中的常见例子
- 对同一把 mutex，先 unlock 再 lock，前面的写对后面的读可见
- channel send happens-before 对应 receive
- close(channel) happens-before 读到关闭状态
- `Once.Do(f)` 中 `f` 的执行结果对后续调用可见

### 面试意义
如果面试官问内存模型，happens-before 基本是绕不过去的核心词。

### 背诵版
> happens-before 表示两个操作之间存在可见性和顺序性保证。Go 并发里只有建立了 happens-before，才能安全地依赖共享数据的结果。

## 12. mutex 和 RWMutex 怎么选？

### 标准答案
一般原则是：

- 共享状态简单、竞争不重：优先 `Mutex`
- 明显读多写少：可考虑 `RWMutex`
- 但不要想当然认为 `RWMutex` 一定更快

### 为什么 `RWMutex` 不总是更优？
因为它实现更复杂，维护读写状态也有成本。
在以下场景中，`RWMutex` 反而不一定占优：
- 临界区很小
- 写操作不少
- CPU 核不多
- 锁竞争本身不重

### 工程建议
除非有明确读多写少特征和实测收益，否则先用 `Mutex`，简单且稳。

### 易错点
- 不要为了"高级"而强行上 `RWMutex`
- 实际性能要用 benchmark 验证

### 背诵版
> `RWMutex` 适合明显读多写少的场景，但它并不一定总比 `Mutex` 快。工程上优先简单方案，必要时通过 benchmark 验证是否值得替换。

<a id="q13"></a>
## 13. 锁竞争严重时怎么排查？

### 标准答案
锁竞争严重通常会表现为：
- CPU 不一定很高，但吞吐下降
- 响应时间上升
- goroutine 大量阻塞
- pprof 中 mutex/block 指标明显

### 排查思路

#### 1）先看现象
- QPS 下降
- RT 上升
- goroutine 堆积
- 某接口超时增多

#### 2）用 pprof
重点看：
- mutex profile
- block profile
- goroutine profile

#### 3）看临界区
- 锁保护范围是否过大
- 锁内是否有 IO、RPC、sleep、复杂计算

#### 4）看数据结构设计
- 是否一个全局大锁保护整个热点 map
- 是否可以分片锁
- 是否可以无锁化一部分读路径
- 是否可以使用 atomic 替代简单状态锁

### 优化方向
- 减小临界区
- 拆锁
- 分片
- 降低共享
- 读写分离
- 缓存热点值
- 消除不必要同步

### 背诵版
> 锁竞争排查主要看 mutex/block/goroutine profile，重点检查是否存在大临界区、锁内 IO、全局热点锁等问题，优化方向一般是缩小临界区、拆锁、分片或减少共享状态。

<a id="q14"></a>
## 14. atomic 和 mutex 的边界在哪里？

### 标准答案
atomic 适合：
- 单个变量的原子读写
- 简单计数器
- 状态位
- 指针替换
- 少量无锁快路径

mutex 适合：
- 多变量一致性维护
- 复合状态变更
- 需要清晰临界区语义
- 修改逻辑较复杂

### 为什么不能滥用 atomic？
因为 atomic 能解决的是"单点原子性"，但解决不了复杂业务语义的一致性。
多字段状态如果强行 atomic，代码会非常难读，也容易出错。

### 面试标准答法
> atomic 适合简单、局部、低级别同步；mutex 更适合复杂状态一致性控制。实际工程里，优先保证正确性和可维护性，不要为了追求无锁而无锁。

### 背诵版
> atomic 解决单变量原子操作，mutex 解决复合状态一致性。简单状态用 atomic，复杂共享状态优先用锁。

<a id="q15"></a>
## 15. sync.Map 适合什么场景？

### 标准答案
`sync.Map` 并不是普通 map 的全场景替代品。
它更适合这些场景：

- 读多写少
- key 相对稳定
- 并发访问频繁
- 缓存类、注册表类结构

### 为什么不是所有场景都适合？
因为 `sync.Map` 内部做了专门优化和分层结构，它适合特定访问模式，但：
- 类型不安全，需要断言
- 复杂业务语义不如 map+lock 清晰
- 写多删多时未必更优

### 工程建议
如果是业务核心数据结构，尤其需要复杂操作，通常还是：
- `map + Mutex/RWMutex`

### 背诵版
> `sync.Map` 更适合读多写少、key 稳定的高并发场景，比如缓存和注册表；如果是复杂业务状态，通常 `map + lock` 更直观也更可控。

<a id="q16"></a>
## 16. 线上 Go 服务 CPU 飙高你怎么排查？

### 标准答案
我一般会按"先确认现象，再定位热点"的思路来排查。

### 第一步：确认范围
- 是单实例还是全量实例
- 是持续飙高还是瞬时抖动
- 是否和流量上涨、发布、依赖异常有关

### 第二步：抓 profile
用 pprof 看：
- CPU profile
- goroutine profile
- mutex/block profile（如果怀疑锁）
- heap profile（如果怀疑 GC 导致 CPU 抖动）

### 第三步：看热点函数
常见方向：
- 死循环 / 自旋
- JSON 编解码过重
- 正则 / 反射耗时
- 锁竞争导致忙等
- GC 压力大
- 热点日志打印
- 大量短连接创建
- 序列化/压缩/加密计算

### 第四步：结合业务链路
- 某个接口是否异常放量
- 某个下游超时是否导致重试风暴
- 某个缓存失效是否导致回源风暴

### 背诵版
> CPU 飙高先确认影响范围和变更背景，再用 pprof 抓 CPU 热点，看是死循环、序列化、锁竞争、GC 压力还是重试风暴等问题，最后再结合业务链路定位根因。

<a id="q17"></a>
## 17. Go 服务内存持续上涨怎么排查？

### 标准答案
先区分是"正常上涨"还是"异常上涨"。

### 可能情况

#### 1）业务数据增长导致堆增长
比如缓存越来越大，这是业务层问题，不一定是泄漏。

#### 2）对象无法释放
比如切片持有大数组、全局 map 不断增长、goroutine 泄漏持有引用。

#### 3）GC 来不及回收 / 分配过猛
导致看起来内存一直高。

### 排查步骤

#### 1）看监控
- heap inuse
- heap alloc
- gc 次数
- goroutine 数
- RSS

#### 2）抓 heap profile
看哪些类型对象占用最多。

#### 3）看对象来源
- 哪些代码路径分配的
- 是否有长期缓存
- 是否有大对象切片
- 是否有未关闭资源引用链

#### 4）检查 goroutine
goroutine 泄漏会持有上下文和对象引用，导致内存回收不了。

### 背诵版
> 内存上涨要先区分是正常业务增长还是异常泄漏，然后结合 heap profile、goroutine profile 和对象分配路径，重点排查缓存膨胀、切片持有大数组、全局引用和 goroutine 泄漏等问题。

<a id="q18"></a>
## 18. goroutine 泄漏线上怎么定位？

### 标准答案
goroutine 泄漏本质是协程没有按预期退出，常见现象包括：

- goroutine 数持续上涨
- 内存缓慢升高
- 某些请求越来越慢
- block profile/goroutine profile 异常

### 常见原因
- channel 永远阻塞
- 下游不消费，生产者卡死
- for-select 没退出分支
- context 没取消
- 网络调用没超时
- worker pool 没有关闭机制

### 定位方法

#### 1）看 goroutine 监控
是否持续增长而不回落。

#### 2）抓 goroutine dump
重点看大量 goroutine 卡在哪些栈帧。

#### 3）分析阻塞点
- chan send
- chan recv
- select
- netpoll wait
- mutex lock

#### 4）追生命周期
看启动点和退出条件是否匹配。

### 背诵版
> goroutine 泄漏定位主要看 goroutine 数量趋势和 goroutine dump，重点分析大量协程阻塞在哪些栈帧，再回到代码检查 channel、context、select 和退出条件是否设计完整。

<a id="q19"></a>
## 19. 高并发服务里 channel 和 mutex 怎么做选择？

### 标准答案
这是一个很常见但很容易回答空泛的问题。
我的理解是：

### 用 channel 的场景
- 任务分发
- 事件通知
- pipeline
- worker pool
- 生产者消费者模型
- 需要通过消息传递来组织流程

### 用 mutex 的场景
- 保护共享状态
- 读写 map、cache、计数结构
- 维护复合对象一致性
- 临界区逻辑明确

### 不要误区化
Go 虽然强调 channel，但不代表所有并发都应该用 channel。
如果只是保护一个共享 map，用 mutex 往往更直接、更高效。

### 面试加分点
> channel 更擅长通信和编排，mutex 更擅长共享状态保护。选型要看问题本质，不是看哪种更"Go 风格"。

### 背诵版
> channel 更适合通信、编排和任务流转，mutex 更适合共享状态保护。工程上不要为了风格强行统一，应该按问题本质选工具。

<a id="q20"></a>
## 20. 高并发架构里 Go 常见模式有哪些？

### 标准答案
Go 在高并发架构里常见模式主要有：

### 1）Worker Pool
限制并发度，控制资源使用。

### 2）Pipeline
将处理拆成多个 stage，通过 channel 串联。

### 3）Fan-out / Fan-in
并行处理多个任务，再汇总结果。

### 4）Context + ErrGroup
一个任务组中任意任务失败，取消其他任务。

### 5）Singleflight
避免缓存击穿或相同请求重复回源。

### 6）Semaphore
限制同类任务最大并发数。

### 7）Actor-like 模式
把共享状态收敛到单 goroutine 内，通过消息操作状态，减少锁。

### 背诵版
> Go 高并发常见模式包括 worker pool、pipeline、fan-out/fan-in、errgroup、singleflight 和 semaphore，本质上都是在解决并发控制、任务编排和资源保护问题。

<a id="q21"></a>
## 21. singleflight 解决什么问题？

### 标准答案
singleflight 用于合并相同 key 的并发请求，避免同一时刻对同一个资源发起重复计算或重复回源。

### 典型场景
- 缓存击穿
- 热点 key 同时失效
- 同一配置并发加载
- 同一用户数据并发刷新

### 效果
多个请求并发进来时，只有一个真正去执行，其他请求等待这个结果复用。

### 易错点
- singleflight 只合并同一个 key 的并发请求
- 它不是缓存本身
- 不能替代过期策略和限流

### 背诵版
> singleflight 用于把同一 key 的并发请求合并成一次执行，常用于防止缓存击穿和热点回源风暴。

<a id="q23"></a>
## 23. 优雅停机一般怎么做？

### 标准答案
优雅停机的目标是：
**停止接新流量，等待存量请求处理完，再安全退出。**

### 常见步骤
1. 收到退出信号，如 SIGTERM
2. 服务实例从注册中心摘除或负载均衡下线
3. 停止接收新请求
4. 对已有请求设置最大等待时间
5. 通知后台 goroutine 停止
6. 关闭连接池、消息消费者、定时任务
7. 等待清理完成后退出

### Go 常见实现
- `signal.NotifyContext`
- `http.Server.Shutdown`
- `context cancel`
- `WaitGroup`
- 任务池退出协议

### 背诵版
> Go 优雅停机一般是收到退出信号后先摘流量，再停止接新请求，并通过 context 和 WaitGroup 等机制等待存量任务收尾，最后关闭资源退出。
