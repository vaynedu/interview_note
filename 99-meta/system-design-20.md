## 1. 系统设计的回答框架？

### 标准答案

**4 步框架**：

```
1. 需求确认（5 分钟）
   - 功能需求（核心 / 次要）
   - 非功能需求（QPS / 数据量 / SLA / 延迟）
   - 边界（不做什么）

2. 容量估算（5 分钟）
   - QPS（峰值 / 日均比）
   - 存储（数据量 × 时间）
   - 带宽（QPS × 平均响应大小）

3. 架构设计（15 分钟）
   - 整体架构图
   - 数据模型（DB 设计）
   - 接口（API 列表）
   - 关键流程

4. 深入细节（10 分钟）
   - 高可用 / 高性能 / 可扩展
   - 难点 / 取舍
   - 监控 / 告警
```

**关键原则**：
- **先简单后复杂**（先单体跑通，再优化）
- **量化思维**（数字驱动而非感觉）
- **取舍意识**（每个决策有成本）

### 易错点
- 不问需求直接画图（搞错方向）
- 没估算就上架构（容量不知道）
- 一开始上微服务（过度设计）

### 追问点
- 估算技巧？→ DAU × 人均行为 × 高峰倍数
- 怎么应对扩展性提问？→ X 轴克隆 / Y 轴拆服务 / Z 轴分片

### 背诵版
**4 步**：需求 → 容量 → 架构 → 深入。**先简单后复杂 + 量化 + 取舍**。开口先问需求和量级。

## 2. 短链系统怎么设计？

### 标准答案

**功能**：
- 长链 → 短链
- 短链访问跳转长链
- 防恶意 / 统计访问量

**核心难点**：
1. **短链生成**：保证唯一 + 短
2. **高可用读**（读 >> 写，10:1）
3. **统计**

**短链生成方案**：
- **发号器 + Base62**：递增 ID（雪花 / 号段）→ Base62 编码（6-7 位）
- **Hash + 冲突解决**：MD5(长链) → 取前 6 位 → 冲突时加盐重试

**架构**：
```
用户 → CDN → 网关 → 短链服务 → Redis（短链 → 长链）→ DB（持久化）
```

**容量估算**：
- 日新增 1 亿短链 → 一年 365 亿
- Base62 6 位 = 56 亿空间（不够）→ 7 位 = 3.5 万亿

**性能优化**：
- Redis 缓存（命中率 90%+）
- 多级缓存（CDN + 本地 + Redis）
- DB 分库分表（按 short_url hash）

**统计**：
- 异步写 Kafka → 离线统计
- Redis Counter 实时

### 易错点
- 用 Hash 不解决冲突
- 不分表（单表撑不住）
- 同步写统计（影响主流程）

### 追问点
- 防恶意？→ URL 黑名单 + WAF + 风控
- 短链失效？→ TTL（短期/永久 + 业务策略）

### 背诵版
**发号器 + Base62 编码（6-7 位）/ Hash + 冲突解决**。**Redis 缓存 + DB 分表**。**异步写统计**。读 >> 写，**多级缓存**。

## 3. 秒杀系统怎么设计？

### 标准答案

**核心难点**：
- 流量高峰（10 万 → 100 万 QPS）
- 库存防超卖
- 防黄牛 / 刷单

**经典 5 层防御**：

```
1. 前端: 静态化 + CDN + 限流（按钮置灰）+ 验证码
2. 接入层: Nginx / 网关 + 限流 + 风控
3. 应用层: 应用集群 + 异步排队 + 限流
4. 缓存层: Redis 库存 + Lua 原子扣减
5. 数据层: 异步落库 + 分库分表
```

**库存方案**：
- **Redis 预热库存**：活动前把库存写 Redis
- **Lua 原子扣减**：`if stock > 0 then stock-- end`
- **异步落库**：扣减成功 → 写 MQ → 后端异步写 DB

```lua
-- 原子扣减库存
local stock = redis.call('get', KEYS[1])
if tonumber(stock) <= 0 then return 0 end
redis.call('decr', KEYS[1])
return 1
```

**防超卖**：
- Redis Lua 原子操作
- DB 层乐观锁兜底（`UPDATE WHERE stock > 0`）
- 唯一索引防重复下单

**防黄牛**：
- 验证码（图形 / 行为）
- 风控（IP 频次 / 设备指纹）
- 限购（每人 N 件）

### 易错点
- DB 直接扣库存（DB 打爆）
- 用 GET + DEL 两步扣（非原子，超卖）
- 同步写订单（响应慢）

### 追问点
- 流量怎么扛？→ CDN + 限流 + 削峰（MQ）
- 大促容量怎么准备？→ T-90 评估 / T-7 全链路压测 / T-1 预热

### 背诵版
**5 层防御**：前端静态化 / 接入层限流 / 应用排队 / **Redis Lua 扣库存 + 异步落 DB** / 风控防黄牛。

## 4. 实时弹幕系统怎么设计？

### 标准答案

**功能**：
- 发弹幕
- 拉取最近弹幕
- 实时推送

**架构**：
```
推送方案选择:
  长轮询 (HTTP)：实现简单，1-3s 延迟
  WebSocket: 实时性强，推荐
  SSE: 单向推送，适合直播弹幕
```

**核心组件**：
- 接入层（WebSocket Gateway）
- 弹幕服务（推送 + 持久化）
- Redis（最近弹幕）
- Kafka（异步落库）
- DB（历史弹幕）

**容量估算**：
- 直播间 100 万人 / 1000 弹幕 QPS
- 推送：100 万 × 1000 = **10 亿次推送/秒** → 必须分流

**优化**：
- **房间分片**：按房间 hash，每台机器维护部分房间
- **批量推送**：100ms 聚合一批推送
- **消息丢弃**：超热房间（如顶流主播），低 QPS 弹幕直接丢
- **本地缓存最近弹幕**

**抗洪峰**：
- 接入层连接池
- 推送批量 + 压缩
- 限流（每用户 N 弹幕/秒）

### 易错点
- 用 HTTP 长轮询做百万房间（连接太多）
- 同步落库（性能差）
- 不丢弃低质弹幕（顶流房间炸）

### 追问点
- WebSocket 连接数怎么扩？→ 长连接网关多实例 + 一致性 Hash 路由
- 历史弹幕怎么按时间查？→ DB 按 video_id + timestamp 索引

### 背诵版
**WebSocket 接入层 + 房间分片 + Redis 缓存 + Kafka 落库 + 批量推送 + 限流丢弃**。百万级房间用长连网关分片。

## 5. 直播系统怎么设计？

### 标准答案

**核心环节**：
- 推流（主播 → 源站）
- 转码（多码率）
- 分发（CDN）
- 拉流（观众）
- 互动（弹幕 / 礼物 / 连麦）

**协议**：
- 推流：RTMP / SRT
- 拉流：HLS（兼容好慢）/ FLV / **LL-HLS**（低延迟）/ **WebRTC**（< 500ms）

**架构**：
```
主播 → 推流 SDK → 源站 → 转码 → CDN 边缘 → 观众

互动:
  弹幕：WebSocket Gateway
  礼物：业务系统 + 钱包
  连麦：WebRTC P2P / SFU
```

**关键优化**：
- **边缘转码**（节省回源带宽）
- **首屏秒开**（关键帧前置 + CDN 预热）
- **抗弱网**（自适应码率 ABR）
- **大主播 1 推百万拉**（树状分发 + CDN 多层）

**容量估算**：
- 顶流 100 万人观看
- 1080p 4Mbps × 100万 = 400 Gbps → 必须 CDN 分担

### 易错点
- 标准 HLS 做电商直播（30s 延迟）
- 不做边缘转码（源站打爆）
- 首屏黑屏（没有关键帧前置）

### 追问点
- 怎么保证连麦低延迟？→ WebRTC < 500ms
- 主播切换 CDN？→ 多 CDN 调度 + 健康检查

### 背诵版
**RTMP 推流 + 转码 + CDN 分发 + LL-HLS/WebRTC 拉流**。**边缘转码 + 首屏秒开 + ABR 自适应**。互动用 WebSocket Gateway。

## 6. Feed 流（信息流）怎么设计？

### 标准答案

**两种主流方案**：

| | 推模式（Push） | 拉模式（Pull） |
| --- | --- | --- |
| 时机 | 发帖时推到所有粉丝 | 用户刷新时聚合 |
| 写性能 | 差（粉丝 1000 万 → 1000 万次写） | 好 |
| 读性能 | 好（直接读 timeline） | 差（实时聚合） |
| 适合 | 普通用户 | 大 V |
| 代表 | Twitter（早期） | Facebook |

**推拉结合（业界主流）**：
- 普通用户：推（写时复制到粉丝 timeline）
- 大 V：拉（读时聚合）

**架构**：
```
用户发帖 → 帖子表 → 粉丝表 → 异步推送到粉丝 timeline（Redis ZSet）

读 timeline:
  Redis ZSet（最近 N 条）
  miss → DB 拉历史
  + 大 V 实时拉取
```

**Redis 数据结构**：
- `user:{uid}:timeline` ZSet（score = 时间戳）

**优化**：
- timeline 限长（保留 1000 条）
- 历史走 DB 分页
- 推荐插入（机器学习）

### 易错点
- 大 V 也用推（百万粉丝写炸）
- 不用 ZSet（无序）
- timeline 不限长（内存爆）

### 追问点
- 怎么平衡时效和性能？→ 推拉结合 + 分级缓存
- 推荐流怎么做？→ 召回（多源）+ 排序（机器学习）

### 背诵版
**推拉结合**：普通推（粉丝 timeline）/ 大 V 拉（读时聚合）。Redis ZSet（score=时间）。**timeline 限长 + 历史走 DB**。

## 7. IM（即时通讯）怎么设计？

### 标准答案

**核心难点**：
- 长连接管理
- 消息可靠性（不丢 / 不重 / 顺序）
- 离线消息
- 群聊（千万级群）

**架构**：
```
Client → WebSocket Gateway（长连接）→ 路由 → IM 服务 → DB / Redis / MQ
```

**消息可靠性 3 层 ACK**：
1. 发送方 → Server（A→S ACK）
2. Server → 接收方（S→B ACK）
3. 接收方处理后 → Server（B→S ACK）

**消息存储**：
- **写扩散**（每用户一份消息）：读快但写多
- **读扩散**（消息只存一份，按会话 ID 拉）：写少读多
- 实战**写扩散主流**（对话型）

**会话表**：
- 双方组合的 session_id（小用户 ID + 大用户 ID）
- 每用户的会话列表（Redis ZSet）

**群聊**：
- 小群（< 100）：写扩散
- 大群（> 1000）：读扩散 + 推送拉取混合

**离线消息**：
- 消息进 MQ + 用户离线状态
- 用户上线 → 拉离线消息

### 易错点
- 不做 ACK（消息丢）
- 大群写扩散（千万级群炸）
- 序列号不连续（顺序乱）

### 追问点
- 怎么保顺序？→ 单会话单分区 + 单消费者
- WebSocket 连接数怎么扩？→ 长连接网关多实例 + 一致性 Hash

### 背诵版
**长连接网关 + 路由 + IM 服务 + 多级 ACK**。小群写扩散 / 大群读扩散。**离线消息 MQ + 上线拉取**。

## 8. 评论系统怎么设计？

### 标准答案

**功能**：
- 发评论 / 删评论
- 评论列表（按时间 / 热度）
- 楼中楼 / 二级评论
- 点赞 / 举报

**数据模型**：
```sql
CREATE TABLE comments (
    id BIGINT,
    target_id BIGINT,        -- 文章/视频 ID
    parent_id BIGINT,        -- 父评论 ID（楼中楼）
    user_id BIGINT,
    content TEXT,
    like_count INT,
    reply_count INT,
    created_at TIMESTAMP,
    INDEX (target_id, created_at)
)
```

**树形结构方案**：
- **闭包表**：所有祖先关系单独表（查询快，写复杂）
- **路径枚举**：path 字段存 `1/2/3`（查询简单）
- **嵌套集合**：lft/rgt（适合读多）
- **限制层级**（实战）：只允许两级（评论 + 回复），简单且够用

**性能优化**：
- 热点评论 Redis 缓存
- 列表分页（cursor 游标）
- 评论数 Redis Counter
- 异步统计（点赞 / 回复数）

**审核**：
- 关键词过滤
- 机器审核（敏感词 / 色情 / 政治）
- 人工抽审

### 易错点
- 树形 N+1 查询（性能差）
- 同步审核（响应慢）
- 不做缓存（DB 打爆）

### 追问点
- 楼中楼怎么实现？→ 限制两级 / 闭包表 / 路径枚举
- 热门评论排序？→ Wilson score / Hot 算法

### 背诵版
**评论表 + parent_id + index**。**限制两级**（评论+回复）。**Redis 热点缓存 + Counter**。**异步审核**。

## 9. 点赞计数系统怎么设计？

### 标准答案

**难点**：
- 高并发（写）
- 准确性（防重复点赞）
- 实时性

**架构**：
```
用户点赞 → Redis（去重 SET + 计数 INCR）→ 异步落 DB
读：Redis 直接返回
```

**Redis 数据结构**：
- `like:{post_id}:users` SET（用户 ID 去重，防重复点赞）
- `like:{post_id}:count` INT（计数）

**写流程**：
```lua
-- Lua 原子去重 + 计数
if redis.call('sismember', users_key, user_id) == 0 then
    redis.call('sadd', users_key, user_id)
    redis.call('incr', count_key)
    return 1
end
return 0
```

**优化**：
- **本地缓存**（应用进程）减 Redis QPS
- **批量落 DB**（每 N 条 / N 秒）
- **冷数据归档**（老帖子 → DB only）
- **HyperLogLog**（粗略统计 UV，省内存）

**防刷**：
- 风控（IP / 设备 / 频次）
- 异常检测

### 易错点
- 用 INCR 不去重（重复点赞）
- 同步落 DB（写性能瓶颈）
- 大量帖子全部 Redis（内存爆）

### 追问点
- 用户量大 SET 内存爆？→ Bitmap（user_id 作 offset）
- HyperLogLog 适合什么？→ 大量 UV 统计，0.81% 误差

### 背诵版
**Redis SET 去重 + INCR 计数 + Lua 原子**。**本地缓存 + 批量落 DB + 冷数据归档**。**Bitmap / HLL** 优化内存。

## 10. 搜索系统怎么设计？

### 标准答案

**核心组件**：
- 索引（倒排索引）
- 查询（QP 解析 + 召回 + 排序）
- 数据同步（实时 / 离线）

**架构**：
```
业务 DB → CDC → Kafka → 索引服务 → Elasticsearch（倒排索引）
                                         ↓
用户 → 网关 → 搜索服务 → ES → 排序 → 返回
```

**倒排索引**：
```
word → [doc_id list]

"苹果" → [doc1, doc5, doc7]
"手机" → [doc1, doc3, doc5]

查询 "苹果 手机" → 求交集 [doc1, doc5]
```

**ES 主流**：分布式倒排索引，水平扩展。

**查询流程**：
1. **Query Parsing**（分词 / 改写 / 纠错）
2. **召回**（多路召回：ES / 个性化模型）
3. **排序**（相关性 + 业务规则 + 机器学习）
4. **聚合 / 过滤**

**优化**：
- 热门 Query 缓存
- 异步索引（DB → Kafka → ES）
- 多副本（ES 默认 3）
- 冷热分层

### 易错点
- 直接查 DB LIKE（性能差）
- 不分词（中文搜索差）
- 同步索引（影响业务）

### 追问点
- 实时性怎么保？→ CDC（Canal / Debezium）订阅 binlog → Kafka → ES
- 个性化怎么做？→ 用户画像 + 召回 + 排序模型

### 背诵版
**ES 倒排索引 + CDC 同步 + 多路召回 + 排序模型**。**Query Parsing + 召回 + 排序 + 聚合**四阶段。

## 11. 文件存储系统怎么设计？

### 标准答案

**功能**：
- 上传 / 下载
- 大文件（GB）
- 高可用 / 高性能

**架构**：
```
Client → CDN → 文件服务 → 元数据 DB（文件信息）+ 对象存储（OSS/S3）

大文件: 分片上传 + 断点续传 + 秒传
```

**关键设计**：

1. **元数据 vs 数据分离**：
   - 元数据：MySQL（文件名 / 大小 / 路径 / hash）
   - 数据：对象存储（S3 / OSS / Minio）

2. **大文件分片上传**：
   - 客户端分 5MB / 10MB 块
   - 并行上传
   - 失败仅重传失败的块
   - 全部上传完触发合并

3. **秒传**：
   - 客户端先算 MD5 / SHA256
   - 服务端检查是否已存在
   - 已存在 → 直接关联，不传

4. **断点续传**：
   - 记录已上传分片
   - 重新连接时跳过已传

5. **去重**：
   - 内容 hash 作为 key
   - 多用户同 hash → 同一份存储 + 多个引用

**容量估算**：
- 日上传 1 亿文件 / 平均 1MB → 100TB/天
- 副本 3 份 → 300TB/天
- 一年 100PB

### 易错点
- 同步上传大文件（卡死）
- 不分片（断网重传整个）
- 不算 hash（无法秒传 / 去重）

### 追问点
- 怎么保高可用？→ 多副本 / 跨机房 / 跨地域
- 冷数据怎么省钱？→ 归档存储（S3 Glacier）

### 背诵版
**元数据 DB + 对象存储**。**分片上传 + 断点续传 + 秒传（hash）+ 去重**。**多副本 + 冷热分层**。

## 12. 支付系统怎么设计？

### 标准答案

**核心要求**：
- 强一致（不能错钱）
- 高可用（不能挂）
- 安全（防伪造）
- 对账

**架构**：
```
业务系统 → 支付网关 → 路由 → 支付渠道（支付宝 / 微信 / 银行）→ 对账系统
                                       ↓
                                  支付订单 DB
```

**关键流程**：

1. **创建支付单**：
   - 业务系统调支付服务
   - 生成支付订单（PENDING）
   - 调用渠道（生成二维码 / 跳转链接）

2. **回调处理**：
   - 渠道回调（验签 + 幂等）
   - 更新支付状态
   - 通知业务系统

3. **状态机**：
   - PENDING → PAID（成功）/ FAILED / CANCELLED / REFUND

4. **对账**：
   - T+1 离线对账（支付 vs 渠道账单）
   - 实时对账（异常事件）

**关键技术**：
- **幂等**（业务请求 ID + 唯一索引）
- **分布式事务**（TCC / Saga）
- **状态机**（严格状态迁移）
- **多渠道路由**（按金额 / 渠道费率 / 可用性）
- **加密签名**（防篡改）

### 易错点
- 不做幂等（重复扣款）
- 同步等渠道回调（响应慢）
- 不对账（金额错没人发现）

### 追问点
- 怎么防双花？→ 唯一业务订单 + 状态机
- 渠道挂了怎么办？→ 自动切换备用渠道 + 排队

### 背诵版
**支付单 + 状态机 + 幂等 + 强一致 + 对账**。**业务订单唯一 + 渠道路由 + 回调验签**。**T+1 对账 + 实时核对**。

## 13. 优惠券系统怎么设计？

### 标准答案

**功能**：
- 发券（推送 / 领取）
- 用券（核销）
- 防刷 / 防重复使用

**数据模型**：
```sql
-- 优惠券模板
CREATE TABLE coupon_templates (
    id BIGINT,
    name VARCHAR,
    type ENUM('FIXED', 'PERCENT'),
    discount DECIMAL,
    total_count INT,           -- 总数
    received_count INT,        -- 已领数
    status ENUM
);

-- 用户优惠券
CREATE TABLE user_coupons (
    id BIGINT,
    user_id BIGINT,
    coupon_id BIGINT,
    status ENUM('UNUSED', 'USED', 'EXPIRED'),
    used_at TIMESTAMP,
    UNIQUE (user_id, coupon_id, ?)  -- 防重复领
);
```

**发券方案**：

1. **预生成**（适合数量少）：
   - 提前生成所有券码到 DB
   - 用户领取时分配一张

2. **实时生成**（适合大批量）：
   - 用户领时生成
   - Redis 计数器控制总量
   - Lua 原子扣减

3. **秒杀型**（高并发领券）：
   - 类似秒杀方案（Redis 库存 + Lua）

**核销**：
- 状态机（UNUSED → USED）
- 用户 + 券 ID 唯一索引（防重复用）
- 与订单事务关联

### 易错点
- 同步扣库存（性能差）
- 不防重复领（用户多领）
- 核销不在事务（用了券订单失败）

### 追问点
- 怎么防黄牛抢券？→ 风控（IP / 设备 / 频次）+ 限领
- 券过期处理？→ 定时任务批量更新状态

### 背诵版
**模板 + 用户券 + 状态机**。秒杀型用 **Redis + Lua 扣库存**。**唯一索引防重复**。**核销在订单事务**。

## 14. 库存系统怎么设计？

### 标准答案

**核心难点**：
- 并发扣减（防超卖）
- 库存预占 vs 实际扣减
- 退货还库存

**两种方案**：

1. **DB 行锁（悲观锁）**：
   ```sql
   UPDATE stock SET count = count - 1 WHERE id = ? AND count >= 1
   ```
   - 优：强一致
   - 缺：性能差，高并发瓶颈

2. **Redis + Lua + 异步落 DB**（高并发）：
   - Redis 预热库存
   - Lua 原子扣减
   - 成功 → 异步写 MQ → 后端落 DB

**库存状态**：
- 总库存
- 已售
- 锁定（已下单未支付）
- 可售 = 总 - 已售 - 锁定

**预占 + 释放**：
```
下单: 预占库存（30 分钟超时）
支付成功: 转为已售
未支付: 自动释放
```

**业务策略**：
- 提前预占（避免超卖）
- 定时释放（30 分钟）
- 退货还库存（原路 + 校验）

### 易错点
- 直接 UPDATE 不带 WHERE count >= 1（超卖）
- 不做预占（用户付款后才扣）
- 退货不做校验（多退）

### 追问点
- 秒杀场景怎么扛？→ Redis Lua + MQ 异步 + 应用层限流
- 多仓库存？→ 仓库分摊 + 路由 + 调拨

### 背诵版
**普通 DB 行锁 / 高并发 Redis Lua + MQ 异步**。**预占 + 定时释放 + 已售**。**WHERE count >= 1** 防超卖。

CREATE TABLE orders (

## 16. 推荐系统架构？

### 背诵版

## 17. 限流系统怎么设计？

### 标准答案

**架构层级**：
```
1. 接入层: Nginx / 网关（粗粒度，全局保护）
2. 应用层: 中间件（业务级，按接口/用户）
3. 中间件: Sentinel / Hystrix / 自研
```

**算法**：
- 固定窗口
- 滑动窗口
- 令牌桶（**主流**）
- 漏桶

**实现**：
- 单机：Guava RateLimiter / sync.atomic
- 分布式：**Redis + Lua**

```lua
-- 令牌桶（Lua）
local tokens = redis.call('get', KEYS[1])
local now = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])
local capacity = tonumber(ARGV[3])

-- 计算新增令牌
local last = redis.call('get', KEYS[2]) or now
local new_tokens = math.min(capacity, (now - last) * rate + tokens)

if new_tokens >= 1 then
    redis.call('set', KEYS[1], new_tokens - 1)
    redis.call('set', KEYS[2], now)
    return 1
end
return 0
```

**多维度**：
- 全局（保护下游）
- 接口级
- 用户级
- IP 级

**配置**：
- 动态配置（配置中心 + 实时推送）
- 灰度规则
- 默认阈值 + 业务覆盖

### 易错点
- 单层限流（被绕过）
- 阈值死板（不分时段）
- 不做监控（限流不生效不知道）

### 追问点
- Sentinel 怎么实现？→ 滑动窗口统计 + 规则引擎 + 实时控制台
- 集群限流怎么算？→ Redis Lua / 基于 token server 集中计数

### 背诵版
**多层（接入 + 应用 + 中间件）+ 多维度（全局/接口/用户/IP）+ 令牌桶 + Redis Lua**。**配置中心动态推送**。

## 18. 配置中心怎么设计？

### 标准答案

**核心组件**：
```
管理控制台 → 配置存储（DB）→ 配置服务 → 客户端 SDK
                                         ↓ 长轮询
                                       业务应用
```

**关键能力**：
- 集中存储 / 动态推送 / 灰度 / 审计 / 加密

**长轮询**（Nacos/Apollo 主流）：
```
Client: 发请求带当前 MD5
Server: 没变化 → 挂住最多 30s → 期间变化立即响应 / 超时返回 304
```

**灰度**：
- 按 IP / 实例 / 比例
- 推送前先验证

**容灾**：
- **客户端本地缓存**（配置中心挂时降级）
- 多副本（DB / 配置服务）

**安全**：
- 配置加密（敏感信息 AES）
- 权限管理（RBAC）
- 审计日志

详见 [../07-microservice/03-config-center.md](../07-microservice/03-config-center.md)。

### 易错点
- 不做客户端缓存（配置中心挂全挂）
- 敏感配置明文（密钥泄漏）
- 改配置不灰度（炸全网）

### 追问点
- Nacos vs Apollo？→ Nacos 简单一体；Apollo 灰度审计强
- 配置数据量大怎么办？→ 分组 + 命名空间隔离

### 背诵版
**长轮询 + 客户端缓存 + 灰度 + 审计 + 加密**。**Nacos 一体（注册+配置）/ Apollo 企业级强审计**。

## 19. 分布式 ID 系统怎么设计？

### 标准答案

**目标**：
- 全局唯一
- 趋势递增（B+ 树插入友好）
- 高性能 / 高可用
- 可分库分表

**主流方案**：

1. **雪花算法**（推荐）：
   ```
   1 bit 符号 + 41 bit 时间戳 + 10 bit 机器 ID + 12 bit 序列号
   ```
   - 优：本地生成性能极高
   - 缺：时钟回拨

2. **号段**（美团 Leaf）：
   - DB 一次取一段（如 1-1000），客户端缓存
   - 用完再取
   - 双 buffer 平滑切换
   - 优：DB 压力低 + 容灾好
   - 缺：实现复杂

3. **Redis INCR**：
   - 简单
   - 单点

**美团 Leaf 双模式**：
- 号段（无时钟问题）
- Snowflake（用 ZK 注册机器 ID 解决时钟回拨）

**生产架构**：
```
应用 → ID 服务（多实例）→ DB / ZK
                ↓
          本地缓存（号段）
```

### 易错点
- UUID 做主键（B+ 树插入分裂）
- 雪花不处理时钟回拨
- ID 服务单点

### 追问点
- 雪花机器 ID 怎么分配？→ ZK 注册 / 配置中心
- 号段双 buffer 怎么做？→ 当前用到 90% 就异步加载下一段

### 背诵版
**雪花算法本地生成 / Leaf 号段无时钟问题**。**ZK 注册机器 ID 解决雪花时钟回拨**。**双 buffer** 平滑切换。

## 20. 全球化系统怎么设计？

### 标准答案

**核心难点**：
- 跨地域延迟
- 多语言 / 多币种 / 多时区
- 合规（GDPR / 数据出境）
- 多地容灾

**架构**：
```
全球 GSLB
    ↓
区域：北美 / 欧洲 / 亚太 / 中国
    ↓
单元化（按用户分单元）
    ↓
本地数据中心
```

**数据策略**：
- **用户数据**：单元化（按 userID hash）
- **元数据**（商品 / 配置）：全球同步（异步复制）
- **强一致数据**（金融）：主单元处理

**多语言 / 多币种**：
- 国际化 i18n（翻译资源）
- 货币换算（实时汇率）
- 时区处理（存 UTC，展示按用户时区）

**合规**：
- GDPR：数据本地化 / 删除权
- 数据出境：跨境流量管控
- 合规节点：欧盟独立 / 中国大陆独立

**全球加速**：
- 全球 CDN / Anycast
- HTTP DNS + HTTP/3
- 跨地域 RPC 优化

### 易错点
- 强一致跨地域（CAP 不允许）
- 不做单元化（跨地域写延迟爆炸）
- 不考虑合规（线上事故）

### 追问点
- 单元化怎么避免跨单元写？→ 按 userID 路由 + 跨单元强一致回主单元
- 时区怎么存？→ DB 存 UTC，展示按用户时区转

### 背诵版
**全球 GSLB + 区域 + 单元化 + 本地数据中心**。**用户数据单元化 / 元数据全球同步**。**国际化 + 时区 + 合规**。
