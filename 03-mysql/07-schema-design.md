# 表设计与架构场景

> 表设计决定了后续索引、事务、扩展和治理成本。MySQL 架构题重点是取舍，不是背一个固定方案。

## 一、核心原理

### 1. 字段类型设计

基本原则：

- 能用小类型不用大类型。
- 能用整数不用字符串。
- 金额用整数分或定点数，不用浮点数。
- 时间字段统一语义，明确时区和精度。
- 字段尽量 `NOT NULL`，减少三值逻辑。
- 大字段和冷热字段可以拆表。

常见选择：

- 主键：`bigint` 或全局 ID。
- 金额：`bigint` 分，或 `decimal`。
- 状态：`tinyint` / `smallint`。
- 时间：`datetime` 或 `timestamp`，团队内统一。
- JSON：适合低频扩展字段，不适合高频过滤字段。

### 2. 约束设计

核心约束要落到数据库：

- 唯一业务键用唯一索引。
- 非空字段用 `NOT NULL`。
- 状态值由应用和数据库共同约束。
- 幂等请求用唯一键兜底。

不要只依赖应用层判断唯一性：

```text
先查有没有 -> 没有再插入
```

这个流程在并发下可能同时插入。正确做法是唯一索引兜底，然后处理唯一冲突。

### 3. 索引设计

索引设计要从查询出发：

- 高频等值过滤字段。
- 范围查询字段。
- 排序字段。
- 分页字段。
- 是否需要覆盖索引。

同时要控制成本：

- 索引越多，写入越慢。
- 索引越宽，占用空间越大。
- 低区分度字段不适合单独建索引。

### 4. 分库分表

分库分表解决的是单库单表容量、写入、连接数、索引膨胀等问题。

一般顺序：

```text
SQL / 索引优化
  -> 冷热归档
  -> 读写分离
  -> 垂直拆分
  -> 水平分库分表
```

分库分表会引入新问题：

- 分布式事务。
- 跨库 join。
- 跨分片排序分页。
- 全局唯一 ID。
- 扩容迁移。
- 数据倾斜。
- 运维复杂度。

所以它不是第一优化手段。

### 5. 线上 DDL

线上 DDL 风险：

- 锁表或阻塞写入。
- 大量 IO。
- 主从延迟。
- 磁盘占用暴涨。
- 执行失败难回滚。

变更前要确认：

- MySQL 版本。
- 表大小。
- 当前写入压力。
- DDL 算法和锁级别。
- 是否影响复制。
- 是否有回滚方案。

## 二、高频面试题

### 如何设计一张订单表？

核心字段：

- 主键 ID。
- 订单号。
- 用户 ID。
- 商户 ID。
- 金额。
- 状态。
- 创建时间、更新时间。
- 支付时间、取消时间等业务时间。

关键设计：

- 订单号唯一索引。
- 用户订单列表索引。
- 商户订单列表索引。
- 状态流转由应用控制，数据库兜底核心约束。
- 金额字段不能用浮点数。
- 历史订单要考虑归档或分区/分表。

示例索引：

```text
uk_order_no(order_no)
idx_user_created(user_id, created_at)
idx_merchant_created(merchant_id, created_at)
idx_status_created(status, created_at)
```

实际是否都需要，要看查询频率和写入压力。

### 如何做幂等？

常见方案：

- 请求唯一号。
- 业务唯一键。
- 数据库唯一索引。
- 状态机判断。

例子：

```text
支付回调：以 payment_no 或 third_trade_no 建唯一索引。
重复回调时插入冲突，转为查询已有处理结果。
```

面试重点：

> 幂等不能只靠 Redis 或应用内存，数据库唯一约束是最后防线。

### 什么时候需要分库分表？

考虑条件：

- 单表数据量过大，索引高度和维护成本明显上升。
- 单库写入压力达到瓶颈。
- 单库连接数、CPU、IO、磁盘容量不足。
- 历史归档、索引优化、读写分离都无法满足增长。

不要只说“超过 500 万行就分表”。阈值和硬件、字段宽度、索引、查询模式有关。

### 分片键怎么选？

好的分片键：

- 高频查询一定带上。
- 分布均匀。
- 不容易变更。
- 能减少跨分片查询。

订单常见分片键：

- 用户侧查询多：`user_id`。
- 商户侧查询多：`merchant_id`。
- 平台按订单号查询多：订单号内嵌分片信息。

难点：

- 用户查自己的订单和商户查自己的订单可能天然冲突。
- 需要通过冗余表、异构索引表、搜索系统解决多维查询。

## 三、典型场景

### 场景 1：用户表手机号唯一

设计：

```text
phone varchar(...) not null
unique key uk_phone(phone)
```

注册流程：

1. 应用校验手机号格式。
2. 插入用户。
3. 如果唯一冲突，返回手机号已注册。

不要：

```text
select count(*) 判断不存在，再 insert
```

并发下会出问题，除非最终仍有唯一索引兜底。

### 场景 2：历史订单越来越大

治理顺序：

1. 明确查询是否主要查最近订单。
2. 建立冷热数据策略，例如近 1 年热表，历史归档表。
3. 高频列表只查热数据，历史查询走归档链路。
4. 报表走离线数仓或汇总表。
5. 如果热数据仍然过大，再考虑水平分表。

不要一开始就分库分表。很多大表问题先通过生命周期治理解决。

### 场景 3：线上误删数据

处理步骤：

1. 先止血，暂停相关写入或入口。
2. 明确误删时间、SQL、影响范围。
3. 使用最近全量备份恢复临时实例。
4. 结合 binlog 恢复到误删前，或跳过误删语句。
5. 导出缺失数据，校验后回灌。
6. 复盘权限、审核、备份演练、延迟从库。

注意：

- 不要在主库上直接盲目反向操作。
- 没演练过的备份风险很高。

### 场景 4：线上加字段

评估：

- 表有多大。
- MySQL 版本是否支持更低成本 DDL。
- 是否有默认值。
- 是否会重建表。
- 是否影响主从复制。
- 是否在高峰期。

保守流程：

1. 先在测试环境验证 DDL 行为。
2. 低峰执行或使用在线 DDL 工具。
3. 观察锁等待、复制延迟、磁盘。
4. 应用代码灰度兼容新旧字段。

## 四、常见坑

- 金额用 float/double。
- 核心唯一性只靠应用判断。
- 字段全部允许 NULL，查询逻辑复杂。
- 状态字段无约束，状态机混乱。
- JSON 字段承载高频查询条件。
- 盲目分库分表，导致跨分片查询更痛苦。
- 分片键不匹配查询路径，最终大量广播查询。
- DDL 前不评估锁和复制延迟。
- 备份从未恢复演练。

## 五、答题模板

### 问表设计

```text
我会先看实体的核心访问路径和生命周期。
字段类型尽量精确，金额不用浮点数，核心字段尽量 not null。
唯一业务约束用唯一索引兜底。
索引围绕高频查询、排序和分页设计，同时控制索引数量。
如果数据有明显冷热特征，会提前设计归档策略。
```

### 问分库分表

```text
分库分表是单库单表到瓶颈后的手段，不是第一选择。
我会先做 SQL 和索引优化、冷热归档、读写分离。
如果容量、写入或连接数仍然成为瓶颈，再水平拆分。
分片键要匹配核心查询路径，并且分布均匀、稳定。
同时要考虑跨分片查询、事务、扩容迁移和全局 ID。
```

### 问线上 DDL

```text
线上 DDL 的风险主要是锁、IO、磁盘、主从延迟和失败回滚。
执行前要确认 MySQL 版本、表大小、语句算法、当前流量和回滚方案。
大表高风险变更可以用在线 DDL 工具，并配合监控和灰度。
```
