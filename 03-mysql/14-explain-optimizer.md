# EXPLAIN 与优化器

> EXPLAIN 不是为了背字段，而是判断 SQL 慢在哪里：扫描多、索引没用好、回表多、排序重，还是优化器选错了计划。

## 一、核心原理

### 1. 优化器做什么

SQL 解析后，优化器会根据统计信息和成本估算选择执行计划：

- 表访问顺序。
- 使用哪个索引。
- 是否全表扫描。
- join 顺序。
- 是否使用临时表或排序。

优化器目标不是“只要有索引就用索引”，而是选择估算成本最低的路径。

```mermaid
flowchart LR
    SQL["SQL"] --> Parse["解析"]
    Parse --> Rewrite["改写"]
    Rewrite --> Stats["读取统计信息"]
    Stats --> Cost["成本估算"]
    Cost --> Plan["选择执行计划"]
    Plan --> Exec["执行"]
```

### 2. EXPLAIN 重点字段

| 字段 | 重点 |
| --- | --- |
| `type` | 访问类型，判断扫描方式 |
| `possible_keys` | 可能使用的索引 |
| `key` | 实际选择的索引 |
| `rows` | 预估扫描行数 |
| `filtered` | 过滤比例 |
| `Extra` | 排序、临时表、覆盖索引等额外信息 |

`type` 大致从好到差：

```text
system / const
  -> eq_ref
  -> ref
  -> range
  -> index
  -> ALL
```

注意：

- `ALL` 不一定绝对有问题，小表全表扫描可能更便宜。
- 大表核心 SQL 出现 `ALL` 通常要重点关注。

### 3. Extra 常见信息

| Extra | 含义 |
| --- | --- |
| `Using index` | 覆盖索引，不需要回表 |
| `Using where` | Server 层还要过滤 |
| `Using filesort` | 需要额外排序 |
| `Using temporary` | 使用临时表 |
| `Using index condition` | ICP 索引条件下推 |

`Using filesort` 不等于一定慢：

- 数据量小可能没问题。
- 有 `limit` 时可能成本可控。
- 但大数据排序、高频接口排序要重点优化。

## 二、高频面试题

### 为什么有索引却没走？

常见原因：

- 索引区分度低。
- 需要回表太多，优化器认为全表扫描更便宜。
- 条件不满足联合索引最左前缀。
- 对索引列用了函数、表达式、隐式类型转换。
- 统计信息不准确。
- 查询返回大量数据。
- `order by`、`group by` 和索引顺序不匹配。

面试表达：

> 有索引不代表一定走索引。优化器根据统计信息估算成本，如果认为全表扫描更便宜，就可能放弃索引。

### rows 很大说明什么？

`rows` 是优化器估算要扫描的行数。

可能说明：

- 索引过滤性差。
- 查询范围太大。
- 统计信息不准。
- 联合索引顺序不合理。

但它是估算值，不是实际扫描值。更严谨可以结合 MySQL 8 的 `EXPLAIN ANALYZE` 或监控实际执行情况。

### possible_keys 有值但 key 为空说明什么？

说明优化器认为有可用索引，但最终没选。

原因可能是：

- 成本估算认为全表扫描更便宜。
- 返回数据比例太高。
- 索引过滤性差。
- 统计信息偏差。

### Using temporary 和 Using filesort 怎么处理？

优化方向：

- 调整联合索引，使过滤和排序尽量在索引上完成。
- 减少排序数据量。
- 减少返回字段。
- 限制时间范围和分页深度。
- 报表类查询迁移到报表库或数仓。

## 三、典型场景

### 场景 1：订单列表不走预期索引

SQL：

```sql
select id, order_no, status, created_at
from orders
where user_id = ?
  and status = ?
order by created_at desc
limit 20;
```

预期索引：

```text
(user_id, status, created_at)
```

如果没走，排查：

- 表数据量是否很小。
- `status` 区分度是否低。
- 查询条件类型是否一致。
- 是否还有其他索引误导优化器。
- 统计信息是否过期。
- 排序方向是否和索引匹配。

### 场景 2：优化器选错索引

现象：

- 同一 SQL 有时快、有时慢。
- 执行计划变化。
- 明明更合适的联合索引没被选。

可能原因：

- 数据分布变化。
- 统计信息不准确。
- 多个索引成本接近。
- 某些参数值过滤性差异大。

处理：

- 更新统计信息。
- 删除重复或误导性索引。
- 调整联合索引。
- 极端情况下使用索引提示，但要谨慎。

### 场景 3：EXPLAIN 看起来还行但线上仍慢

可能原因：

- EXPLAIN 是估算，不是实际执行。
- 慢在锁等待。
- 慢在回表 IO。
- 慢在网络传输。
- 慢在连接池等待。
- 参数不同导致执行计划不同。

所以要结合：

- 慢查询日志。
- 实际执行耗时。
- 锁等待。
- rows examined。
- 应用链路耗时。

## 四、常见坑

- 只看 `key`，不看 `rows` 和 `Extra`。
- 看到 `Using filesort` 就认为一定有问题。
- 看到走索引就认为一定快。
- 忽略隐式类型转换。
- 忽略统计信息和数据分布。
- 用 `force index` 硬压优化器，后续数据变化后反而更慢。
- 只在测试库 EXPLAIN，不看生产数据分布。

## 五、答题模板

```text
EXPLAIN 我重点看 type、key、rows、filtered 和 Extra。
先判断访问方式，是 ref、range 还是全表扫描；
再看实际用了哪个索引，预估扫描多少行；
最后看是否有 Using filesort、Using temporary、Using index。
如果有索引没走，我会从最左前缀、函数表达式、隐式转换、区分度、回表成本和统计信息几个方向排查。
```
