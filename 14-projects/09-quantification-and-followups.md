# 量化指标库 + 追问应对手册

> 面试时"**数字**"和"**追问**"是两道坎。没有量化 → 空洞；答不住追问 → 露底。
>
> 本篇提供：**行业典型指标参考**（让你的数字有依据）+ **50+ 追问应对预案**（应付深挖）。
>
> 与 [06-interview-star-template.md](06-interview-star-template.md) 配合使用。

---

## 一、为什么需要量化

```
低级答法（模糊）:
  "性能提升了很多"
  "撑住了大促"
  "用户体验好了"
  → 听不出干了什么

高级答法（量化）:
  "P99 从 2s 降到 150ms（13 倍）"
  "大促峰值 12 万 QPS，零 P0 事故"
  "客诉减少 80%"
  → 立刻有说服力
```

**量化的价值**：

```
✓ 说服力（数字胜千言）
✓ 边界清晰（多大规模的改进）
✓ 对比参照（是不是有意义的改进）
✓ 追问切口（面试官可以深挖）
✓ 体现数感（好工程师必备）
```

## 二、面试必备：你应该知道的业务数据

### 2.1 自我介绍前必须背下的 5 组数字

```
组 1: 业务规模
  □ 日活 / 月活（DAU / MAU）
  □ 核心操作 QPS（订单 / 消息 / 请求）
  □ 数据量（GB / TB）

组 2: 系统容量
  □ 机器数量（应用 / DB / Redis / MQ）
  □ 单实例规格（CPU / 内存 / 网卡）
  □ 峰值 / 日常对比

组 3: 性能指标
  □ P50 / P95 / P99（单位 ms）
  □ 吞吐（QPS / TPS）
  □ 错误率 / 可用性（SLA）

组 4: 成本
  □ 机器成本（年 / 月）
  □ 带宽 / 存储
  □ 人员成本

组 5: 业务价值
  □ GMV / 收入
  □ 转化率 / 留存
  □ NPS / 客诉
```

**准备清单**：

```
□ 每个数字都真实（不瞎编）
□ 大致范围清楚（不精确也行）
□ "之前 → 之后"对比（体现改进）
□ 跟行业对标（显得有数感）
```

## 三、行业典型指标参考（帮你校准）

### 3.1 Web 服务性能

| 指标 | 优秀 | 一般 | 差 |
| --- | --- | --- | --- |
| API P99 | < 100ms | 100-500ms | > 1s |
| API P999 | < 500ms | 500ms-2s | > 5s |
| 错误率 | < 0.01% | 0.01-0.1% | > 0.1% |
| 可用性 | 99.99%+ | 99.9-99.99% | < 99.9% |
| 吞吐（单核）| > 5000 QPS | 1000-5000 | < 1000 |

### 3.2 数据库 (MySQL)

| 指标 | 优秀 | 一般 | 差 |
| --- | --- | --- | --- |
| 单库 QPS | > 10000 | 3000-10000 | < 3000 |
| 主从延迟 | < 100ms | 100ms-1s | > 1s |
| Buffer Pool 命中率 | > 99.5% | 98-99% | < 98% |
| 慢查询 / 天 | < 100 | 100-1000 | > 1000 |
| 单表行数 | < 500w | 500w-2000w | > 5000w |

### 3.3 Redis

| 指标 | 优秀 | 一般 | 差 |
| --- | --- | --- | --- |
| 单节点 QPS | > 80000 | 50000-80000 | < 50000 |
| P99 | < 1ms | 1-5ms | > 10ms |
| 命中率 | > 95% | 90-95% | < 90% |
| 内存碎片率 | < 1.5 | 1.5-2 | > 2 |
| 连接数 | < 5000 | 5000-8000 | > 8000 |

### 3.4 消息队列 (Kafka)

| 指标 | 优秀 | 一般 | 差 |
| --- | --- | --- | --- |
| 单 broker 吞吐 | > 500MB/s | 100-500MB/s | < 100MB/s |
| 消费延迟 (Lag) | < 1000 | 1000-10000 | > 100000 |
| 生产 P99 | < 50ms | 50-200ms | > 500ms |
| 分区数 | 10-100 | 根据业务 | > 1000 单 topic |

## 四、各场景量化模板

### 4.1 性能优化项目

```
必须有的数字:
  □ QPS 提升: X → Y (N 倍)
  □ RT 降低:  P99 X ms → Y ms
  □ 机器节省: X 台 → Y 台
  □ 成本节省: X 万/年

模板:
  "优化前:
   - P99 = 500ms
   - QPS = 2k
   - 机器 = 50 台

   优化后:
   - P99 = 50ms (10 倍)
   - QPS = 20k (10 倍)
   - 机器 = 10 台
   - 成本节省: 40 台 × 1w/月 = 40w/月 = 480w/年"
```

### 4.2 高并发项目（秒杀 / 大促）

```
必须有的数字:
  □ 峰值 QPS
  □ 持续时间
  □ 参与用户数
  □ 业务成交量
  □ 零事故/小事故次数

模板:
  "双 11 大促:
   - 峰值 QPS: 12 万（预估 10 万，+20%）
   - 持续: 24 小时
   - 参与用户: 3000 万
   - GMV: 5 亿（YoY +30%）
   - P0 事故: 0
   - P1 事故: 2（快速恢复，< 10min）
   - 客诉: < 0.01%"
```

### 4.3 架构改造项目

```
必须有的数字:
  □ 改造前/后架构规模（服务数 / 表数 / 代码行）
  □ 开发效率提升（部署时间 / 迭代周期）
  □ 稳定性提升（MTBF / MTTR）
  □ 成本变化（机器 / 人力）

模板:
  "单体 → 微服务:
   - 代码行: 50w → 7 个服务 × 3-5w
   - 部署时间: 30min → 3min (10 倍)
   - 迭代周期: 2 周 → 1 周
   - 故障恢复: 1h → 15min (4 倍)
   - 单次发布影响: 全局 → 单服务
   - 机器使用率: 20% → 60%（容器化）"
```

### 4.4 故障复盘项目

```
必须有的数字:
  □ 故障时长
  □ 影响范围（用户数 / 请求数）
  □ 业务损失（GMV / 客诉）
  □ 恢复时间（发现 → 止血 → 完全恢复）
  □ 改进后效果

模板:
  "2024 年某次 DB 故障:
   - 故障时长: 45 分钟
   - 影响请求: 500 万
   - 影响用户: 50 万
   - GMV 损失: 100 万
   - 发现时间: 5 分钟
   - 止血时间: 15 分钟
   - 完全恢复: 45 分钟

   改进后:
   - 类似故障 0 复发（1 年）
   - 平均 MTTR 从 1h → 15min
   - 监控覆盖率 60% → 95%"
```

### 4.5 可靠性改进

```
必须有的数字:
  □ 错误率降低（绝对值 + 相对值）
  □ 一致性指标
  □ 数据准确性
  □ 审计覆盖率

模板:
  "订单一致性治理:
   - 不一致率: 0.01% → 0.0001%（100 倍）
   - 月度资损: 120 万 → 1000 元（1200 倍）
   - 实时告警覆盖: 5 分钟发现
   - 自动修复率: 90%（剩 10% 人工）
   - 财务对账时间: 4h/天 → 30min/天"
```

### 4.6 成本优化

```
必须有的数字:
  □ 资源减少（机器 / 带宽 / 存储）
  □ 成本节省（绝对值 + 百分比）
  □ 性能是否降级

模板:
  "CDN 成本优化:
   - 带宽: 10Tbps → 6Tbps (-40%)
   - 命中率: 85% → 95%
   - 年成本: 1000w → 600w（-400w）
   - P99 RT: 无变化
   - 用户体验: 无感知"
```

## 五、50+ 面试追问应对预案

### 5.1 业务类追问（10 个）

```
Q1: "业务规模到底多大？"
A: 给 3 组数字（DAU / QPS / 数据量）
  "日活 500 万，核心接口日常 5k QPS / 大促 10w QPS，
   数据库数据量 500GB（订单表 5 亿行）"

Q2: "你们的核心指标是什么？"
A: 分 3 层讲
  "技术指标: P99 / 可用性 / 错误率
   业务指标: GMV / 转化率 / 留存
   团队指标: MTTR / 迭代速度"

Q3: "你对业务了解多少？"
A: 讲业务模式 + 用户画像 + 商业逻辑
  "我们是 B2C 垂类电商，用户是白领女性，
   客单价 300 元，复购率 30%，
   核心 KPI 是 GMV 和 DAU"

Q4: "你做的这事业务价值是什么？"
A: 把技术改进转为业务数字
  "P99 降 10 倍 → 用户流失率降 5% → GMV +2%"

Q5: "竞品怎么做的？"
A: 至少了解 1-2 个竞品
  "淘宝是 X 方案，京东是 Y，
   我们因为业务特点 Z 选了 A 方案"

Q6: "这个功能 ROI 怎么算？"
A: 投入（人力 + 资源）vs 产出（业务 + 成本）
  "3 个人 3 个月 = 9 人月 = 45w 人力
   收益: 成本节省 400w/年 + GMV +2%
   → ROI 10x"

Q7: "为什么优先做这个？"
A: 痛点 + 紧迫 + 价值
  "因为 [痛点] 影响 [核心指标]，
   每季度损失 X，不做来不及"

Q8: "上线后老板反馈？"
A: 正面 + 具体
  "老板拉会表扬了两次，
   后续给更多资源"

Q9: "用户反馈？"
A: 数据 + 具体案例
  "NPS 从 7 → 9，
   某大 V 用户主动写了感谢"

Q10: "商业化影响？"
A: 直接关联收入
  "直接带动 GMV + 2%，
   间接让公司融到 A 轮"
```

### 5.2 技术深度类追问（15 个）

```
Q11: "为什么选 Redis 不选 MongoDB？"
A: 按场景对比 3 维度
  "访问模式: 我们是 KV + TTL + 排行 → Redis 贴合
   性能: 内存访问 < 1ms
   团队: 已有 Redis 运维，避免新增"

Q12: "Redis 集群怎么部署？"
A: 拓扑 + 规格 + 参数
  "Cluster 模式 6 节点（3 主 3 从）
   每节点 16GB 内存
   maxmemory = 12GB
   淘汰策略 allkeys-lfu"

Q13: "慢 SQL 怎么发现？"
A: 工具链完整
  "slow log + pt-query-digest 聚合
   + sys schema
   + 应用侧 OpenTelemetry 打点
   + 告警（> 1s 立即）"

Q14: "怎么保证消息不丢？"
A: 生产 + 存储 + 消费 3 层
  "Producer: acks=all + 重试
   Broker: RF=3 + ISR
   Consumer: 手动 commit + 幂等"

Q15: "锁选 Redis 还是 etcd？"
A: 按一致性要求分
  "业务幂等可兜底 → Redis（快）
   金融 / 资损 → etcd（强一致）
   我们大部分场景用 Redis + Fencing Token"

Q16: "怎么做容量规划？"
A: 完整流程
  "业务预估 → 压测 → 扩容 → 预案
   双 11: T-90 评估 / T-30 扩容 / T-14 压测"

Q17: "熔断阈值怎么定？"
A: 数据驱动
  "历史 P99 + 业务容忍度
   错误率 > 50% 10s 内 → 熔断
   半开时允许 3 个请求探测"

Q18: "你们的 SLA 是什么？"
A: 具体数字 + 达成情况
  "SLA 99.95%（年 4.4h 不可用）
   SLO 99.99%（内部目标）
   实际过去一年达成 99.98%"

Q19: "监控覆盖率怎么算？"
A: 多层覆盖
  "应用层: RED（请求/错误/耗时）100% 覆盖
   资源层: USE（利用率/饱和/错误）90%
   业务层: 核心指标 100%"

Q20: "分库分表怎么选分片键？"
A: 3 维度权衡
  "80% 查询带用户 ID → user_id
   其他 20% 用反向索引 / 数仓兜底
   选了最优解 + 可接受代价"

Q21: "Kafka 分区数怎么定？"
A: 吞吐 / 并发 / 顺序
  "吞吐 × 消费延迟 = 分区数
   加 20% 冗余
   单 topic < 1000 分区"

Q22: "索引怎么设计？"
A: 覆盖索引 + 最左前缀
  "联合索引: 等值在前 / 范围在后 / 高选择性优先
   尽量覆盖索引（避免回表）
   EXPLAIN 验证"

Q23: "缓存一致性怎么讲？"
A: 方案对比 + 选型
  "4 种方案:
   先写 DB 再删缓存（主流）
   延迟双删
   binlog 订阅
   强一致读
   按一致性要求选"

Q24: "读写分离遇到主从延迟怎么办？"
A: 多档方案
  "强一致读 → 强制主
   写后读 → 缓存兜底
   监控延迟 → 超阈值降级"

Q25: "事务嵌套怎么处理？"
A: Save Point
  "外层事务 + 内层 Save Point
   内层失败只回滚到 save point
   外层继续"
```

### 5.3 故障 / 应急类追问（10 个）

**Q26: "讲一次严重故障"**

A: STAR + 5 Whys 完整 — 见 [07-ecommerce-story.md 故事 4](07-ecommerce-story.md)

**Q27: "故障当时怎么定位的？"**

A: 监控 + 工具 + 路径
- 监控大盘看症状
- 链路追踪定位服务
- 日志看错误栈
- DB 排查 / Redis 排查
- 应用 pprof

**Q28: "怎么止血"**

A: 限流 / 降级 / 切流 / 回滚 4 招
1. 限流保护
2. 非核心降级
3. 切流备用
4. 紧急回滚

**Q29: "恢复后怎么复盘？"**

A: STAR + 5 Whys + 改进项 — 时间线 → 根因 → 改进 + Owner + DDL，全公司共享

**Q30: "类似故障怎么防？"**

A: 监控 + 演练 + 流程 + 工具（加监控 / 季度演练 / 补流程 / 工具自动化）

**Q31: "P0 怎么定义？"**

A: 分级标准
- P0: 核心业务中断 > 5 分钟 / 用户 > 10w
- P1: 核心降级 / 5-30 分钟
- P2: 非核心
- P3: 小 bug

**Q32: "MTTR 怎么降低？"**

A: 3 个阶段优化
- 发现快：监控 + 告警 + AIOps
- 定位快：链路追踪 + Runbook
- 恢复快：自动化 + 预案

**Q33: "一次误删数据怎么处理？"**

A: 按 [03-mysql/23-antipatterns-dangerous-ops.md 案例 1](../03-mysql/23-antipatterns-dangerous-ops.md)

**Q34: "Redis 挂了业务怎么办？"**

A: 熔断 + fallback
- 熔断触发 → 走 DB（限流保护）
- 或返回默认值（降级）
- 业务不能 100% 依赖 Redis

**Q35: "讲一次你救了火的经历"**

A: STAR 具体案例 + 你的关键作用
- 半夜 3 点电话告警
- 5 分钟定位是 X
- 10 分钟止血
- 2h 完全恢复
- 我关键作用：Y

### 5.5 成长 / 反思类追问（10 个）

- 长期：成为技术专家 / 技术管理

**Q51: "你的缺点？"**

A: 真实 + 改进
- 过去比较关注技术，对业务不够深入
- 最近 1 年刻意跟产品 / 运营沟通
- 成长明显

**Q52: "你的优势？"**

A: 3-5 个具体
- 技术扎实（Go + 分布式 + 架构）
- 业务理解（电商 3 年）
- 带团队（4-5 人）
- 救火能力（多次 P0 处理）

**Q53: "为什么离职？"**

A: 正面 + 寻求
- 当前公司业务阶段稳定
- 想找更有挑战的
- 贵司的 X 业务 / Y 技术吸引我

**Q54: "为什么选我们？"**

A: 3 理由
- 业务：做 X，和我经验匹配
- 技术：Y 栈，我感兴趣
- 团队：Z 风格，我适合

**Q55: "你想问什么？"**

A: 2-3 个有质量的问题
- 团队现在最大的技术挑战是什么？
- 我加入后前 3 个月希望做什么？
- 这个岗位过去 1 年的 TopN 成果是什么？

## 六、应对 "不知道" 的高级技巧

### 6.1 承认 + 拓展

```
❌ 简单："这个我不会"
✓ 拓展："这块我当时没深入做，
         但我了解思路是 X，
         可能的方案是 Y，
         你看这样理解对不对？"
```

### 6.2 类比 + 迁移

```
面试官: "你们用过 Dubbo 吗？"
你没用过 Dubbo 但用过 gRPC

✓ "Dubbo 我没用过，
    但我们用 gRPC。
    两者都是 RPC 框架，
    Dubbo 社区生态更广（Spring Cloud Alibaba），
    gRPC 跨语言更好。
    核心思路都是 service + proto + 负载均衡。
    如果切过去应该很快上手。"
```

### 6.3 反问 + 学习

```
✓ "这个我确实不熟。
    您愿意分享一下思路吗？
    我想学习一下。"

→ 把问题转成交流
→ 体现学习能力
→ 面试官通常会讲，反而拉近关系
```

### 6.4 框架化应对

```
即使不会细节，给个框架:

"系统设计题我一般按 5 步走:
 1. 需求确认
 2. 容量估算
 3. 整体架构
 4. 关键取舍
 5. 难点风险

 您想让我按这个顺序展开吗？"

→ 专业感 + 不空口答
```

## 七、面试官的"坑题"识别

### 7.1 你在吹牛吗？

```
面试官追问:
  "你说这事你主导的，
   那 X 细节怎么做的？"
  "你说 10w QPS，
   压测怎么做的？用啥工具？"

应对:
  真做过 → 自然答
  没做过 → 诚实承认 + 说清自己参与程度
```

### 7.2 你有原则吗？

```
面试官："如果老板让你做 X，你觉得不合理，怎么办？"

❌ "老板说做就做"（没原则）
❌ "我会拒绝"（太刚）
✓ "先了解老板的意图
    讲我的担心和数据
    给备选方案
    老板坚持就推着做，留好兜底"
```

### 7.3 你懂业务吗？

```
面试官："你们业务的北极星指标是什么？"
面试官："为什么这个指标重要？"

应对:
  准备好业务 KPI
  讲清指标 + 背后商业逻辑
```

### 7.4 你能反思吗？

```
面试官："如果重做你会怎么改？"

❌ "做得挺好的"
✓ "反思 3 点:
    1. 架构上 X 有隐患
    2. 流程上 Y 没做好
    3. 团队 Z 改进空间"
```

## 八、通用量化话术模板

### 8.1 "提升了多少"

```
❌ "性能提升了很多"
✓ "性能提升了 10 倍（P99 500ms → 50ms）"

❌ "用户体验变好"
✓ "首屏时间从 3s → 1s，转化率 +5%"
```

### 8.2 "节省了多少"

```
❌ "节省了成本"
✓ "成本节省 400w/年（机器 40 台 × 1w/月）"

❌ "节省了时间"
✓ "部署时间 30min → 3min，每天 20 次部署节省 9h"
```

### 8.3 "影响了多少"

```
❌ "影响不少用户"
✓ "影响 500w 用户 / 45 分钟，GMV 损失 100w"

❌ "可用性提升"
✓ "可用性 99.9% → 99.99%，年宕机时间 8.76h → 52min"
```

### 8.4 "规模多大"

```
❌ "挺大规模"
✓ "日订单 200w / 峰值 10w QPS / 数据 500GB"

❌ "很多用户"
✓ "DAU 500w / MAU 2000w"
```

## 九、数字的"真实感"技巧

```
✓ 给范围（不给精确）:
  "QPS 大概 8-10w 左右"
  "P99 在 100-200ms 之间"

✓ 给对比（相对值）:
  "比之前降了 10 倍"
  "比行业平均快 2 倍"

✓ 给依据（怎么得到的）:
  "从监控系统看过去 7 天 P99 的平均"
  "压测工具跑出来的"

✓ 承认不确定（诚实）:
  "具体数字记不清了，范围在 X-Y"
  "那块是 DBA 看的，我大概记得 Z"

✗ 避免伪精确:
  "QPS 10234"（太假了）
  "P99 137.42ms"（显得在编）
```

## 十、面试加分点

- **每个项目都有 5+ 组数字**（业务 + 技术 + 成本 + 价值）
- **范围 + 对比 + 依据** 数字真实感
- **追问预案 50+** 深度应对
- **承认不会 + 拓展思路** 高级技巧
- **识别"坑题"** 防踩雷
- **对比行业指标** 显示数感
- **讲失败 + 反思** 真实可信
- **业务 + 技术 + 团队 + 成长** 4 维度兼顾

## 十一、面试前 checklist

```
□ 背下自己项目的 5 组核心数字
□ 对比行业典型指标（校准自己）
□ 至少准备 3 个项目故事（STAR-L）
□ 每个故事 5-10 个"小细节"可以展开
□ 50+ 追问预案过一遍
□ "不知道"怎么应对练 3 遍
□ 失败经历准备 1-2 个（真实 + 反思）
□ 最骄傲 / 最难 / 最大成长 各一个
□ 反问问题准备 3 个
□ 录音 Mock 1-2 次
```
