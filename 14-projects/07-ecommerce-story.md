### 故事 1：分库分表（架构演进）

**Situation**：

```
2023 接手时:
  - 订单表 5000w 行
  - 单库 MySQL
  - 慢查询频发
  - 大促必挂
  - 备份要 3 小时
```

**Task**：

```
3 个月内完成订单库分库分表
  + 数据迁移
  + 业务零感知（不能停服）
  + 容量目标: 撑 5 年（按业务增速）
```

**Action**：

```
方案对比:
  A. TiDB
     ✓ 自动分片 / 强一致
     ✗ 团队不熟 / 性能 P99 不如 MySQL / 运维复杂
     → 否

  B. Vitess
     ✓ 在线分片 / 兼容 MySQL
     ✗ 国内案例少 / 学习成本高
     → 否

  C. 自研中间件 + 分库分表（最终选）
     ✓ 团队熟 / 可控 / 性能好
     ✗ 自己运维
     ✓ 配合 ShardingSphere 减少自研

最终方案:
  - 16 库 × 8 表 = 128 个物理表
  - 分片键: user_id（80% 查询带 user_id）
  - 算法: user_id Hash → slot → 库表
  - 中间件: ShardingSphere-Proxy
  - 订单号: 雪花算法（含 user_id 段反向解析）

实施 5 步:
  1. 双写阶段（2 周）
     新代码同时写老库 + 新库
     读仍然走老库
     校验数据一致性
  
  2. 历史数据迁移（4 周）
     按时间窗口分批迁移
     增量靠 binlog（Canal）追平
  
  3. 灰度切流（2 周）
     按 user_id 灰度（先 1% → 10% → 50% → 100%）
     有问题秒级回退
  
  4. 老库下线（1 周）
     观察 1 周
     双写关闭
     老库归档
  
  5. 监控完善（持续）
     分库慢查询监控
     单库容量告警
     数据倾斜检测

技术细节:
  - 分布式 ID: 雪花算法
    - 41 bit 时间戳 + 10 bit 机器 ID + 12 bit 序列
    - 时钟回拨: 监控 + 异常报警
  
  - 跨库查询:
    - 80% 带 user_id → 单库
    - 商家查订单 → 反向索引（merchant_id → 订单号列表 存 ES）
    - 报表查询 → binlog → ClickHouse
  
  - 跨库事务:
    - 订单 + 库存: TCC（Seata）
    - 订单 + 通知: Outbox + Kafka
  
  - 扩容:
    - 一致性 Hash（虚拟节点）
    - 未来可平滑扩容
```

**Result**：

```
量化结果:
  - 单表 5000w → 单表 50w（缩 100 倍）
  - 慢查询 -95%
  - DB CPU 高峰 80% → 30%
  - 备份时间 3h → 20min
  - 撑住后续 3 年业务增长

里程碑:
  - 项目周期 3 个月
  - 业务零中断
  - 数据零丢失
```

**Learnings**：

```
1. 反思 - 分片键选错的隐患:
   一开始考虑过用 order_id 分片，
   后来发现 80% 查询带 user_id，改成 user_id。
   现在的痛点: 商家查订单要走 ES 反向索引，
   维护成本高。
   如果重做会上 TiDB 简化（5 年后 TiDB 更成熟）

2. 反思 - 灰度阶段保守不够:
   1% 灰度时遇到一个 corner case
   导致部分订单写入异常
   还好双写兜底。
   下次会加更细的验证（行级 diff）

3. 团队成长:
   团队第一次做大数据迁移
   定了 5 步法 + checklist
   后来推到全公司其他业务
```

### 故事 2：大促 10w QPS 防超卖

**Situation**：

```
2024 年双 11 前压测:
  - 平时 5k QPS 没问题
  - 压到 10w QPS:
    - DB 主库 100% CPU
    - Redis 单分片 100% CPU
    - 出现超卖（库存 -3）
```

**Task**：

```
1 个月内:
  - 撑住 10w QPS
  - 防超卖 100%
  - P99 < 500ms
  - 业务无感知
```

**Action**：

```
5 层防御架构:

L1 CDN + 静态化:
  商品详情页 + 倒计时进 CDN
  按钮在开抢前置灰
  → 80% 流量 CDN 截止

L2 网关 + 限流 + 风控:
  APISIX 整体限流 10w QPS
  用户级限流 1 user/s
  IP 限流 10/s
  风控识别黄牛设备 + 验证码
  → 黄牛拦截 50%

L3 应用层:
  100 pod K8s 集群
  本地令牌桶限流
  漏桶排队
  → 流量整形

L4 Redis Lua（核心防超卖）:
  库存预热到 Redis
  Lua 原子扣减 + Set 去重:
    local stock = redis.call('GET', KEYS[1])
    if not stock or tonumber(stock) <= 0 then
        return 0
    end
    if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
        return -1  -- 已抢过
    end
    redis.call('DECR', KEYS[1])
    redis.call('SADD', KEYS[2], ARGV[1])
    return 1
  
  Hash Tag 让 stock 和 user-set 同 slot:
    {item:123}:stock
    {item:123}:users

L5 MQ 异步落库:
  Redis 抢到 → Kafka 发订单事件
  Worker 异步消费 → 落 DB
  DB 按 user_id 分库（写入分散）
  失败重试 + 死信

L6 DB 兜底:
  CAS 乐观锁:
    UPDATE stock SET count=count-1, version=version+1
    WHERE item_id=? AND count>0 AND version=?

热 key 处理:
  超热商品 → 拆 key（item:123 → item:123:slot0~9）
  本地缓存 5s TTL（接受短期不一致）

降级开关:
  L1: 关闭推荐 / 评论
  L2: 商品详情走兜底缓存
  L3: 只 VIP 抢
  L4: 完全关闭抢购（保护下游）
```

**Result**：

```
压测验证:
  - 10w QPS 撑住，P99 = 280ms
  - 超卖测试: 0
  - 重复抢测试: 0

双 11 实际:
  - 峰值 12w QPS（超预期）
  - 无超卖
  - 无 P0 事故
  - GMV 增长 30%（YoY）

复盘亮点:
  - 5 层防御每层独立可降级
  - Lua 脚本保证原子
  - MQ 削峰让 DB 平稳
```

**Learnings**：

```
1. 反思 - 单分片热点的死角:
   超热商品（爆款）单 slot CPU 100%
   后来加 key 分片 + 本地缓存才解决
   
   教训: 不要假设 Redis Cluster 能扛所有热点
   单分片仍有 12w QPS 极限

2. 反思 - 降级演练不够:
   双 11 前演练降级开关
   发现 L4 开关有 bug（影响 VIP 也无法抢）
   还好提前发现
   
   教训: 降级 = 真演练 + 全链路验证
   不能纸上谈兵

3. 团队收获:
   建立"5 层防御"模板
   后来推到秒杀 / 拼团 / 抽奖
```

### 故事 3：订单一致性对账

**Situation**：

```
现象:
  财务每天跑 T+1 对账
  发现 0.01% 订单状态不一致
    - DB 显示已支付
    - 实际未到账
    - 或反过来

  绝对数: 200 万订单 × 0.01% = 200 单/天
  按客单价 200 元 = 4 万元/天 资损
```

**Task**：

```
彻底解决一致性问题
  + 实时对账（不能 T+1 才发现）
  + 自动修复（不能靠人工）
  + 月度损失 < 1000 元
```

**Action**：

```
根因排查:
  - 网络抖动: 订单服务 → 支付回调超时
  - 重试无幂等: 重复扣款
  - DB 主从切换: 部分写丢失
  - 第三方支付通道: 异步回调延迟

3 层方案:

L1 业务幂等（治根）:
  - 订单号 + DB 唯一索引
  - 状态机严格（PENDING → PAID 不可逆）
  - 支付回调用幂等 token（payment_no）
  - 所有写操作: UPDATE WHERE status=expected

L2 实时对账（监控）:
  - 订单状态变更 → Kafka
  - 支付状态变更 → Kafka
  - Flink 流处理 join 两个流
  - 5 分钟窗口对账
  - 不一致 → 立即告警

L3 T+1 全量对账（兜底）:
  - 订单库 + 支付库 + 第三方对账文件
  - 三方比对（订单号 + 金额 + 状态）
  - 不一致自动进修复队列
  - 修复任务带审计 + 二次确认

修复流程（自动 + 人工）:
  - 90% 不一致 → 规则引擎自动修复
    - 已支付未更新订单 → 补更新
    - 已退款未到账 → 通知财务
  - 10% 复杂 → 进人工队列
  - 每周复盘 → 反向优化业务
```

**Result**：

```
量化结果:
  - 不一致从 0.01% 降到 0.0001%
  - 月度损失从 120 万 → 1000 元以内
  - 实时告警 5 分钟内发现
  - 90% 不一致自动修复
  - 财务从每天对账 4 小时 → 30 分钟

业务价值:
  - 资损减少 99%
  - 用户投诉降 80%
  - 财务效率提升 8 倍
```

**Learnings**：

```
1. 反思 - 一致性是设计问题不是补救:
   一开始我们想靠对账兜底
   后来发现治标不治本
   从根上加幂等 + 状态机才是关键
   
   教训: 一致性要在设计阶段考虑
   补救成本 10x

2. 反思 - 实时对账价值:
   T+1 对账等到第二天才知道
   实时对账 5 分钟发现
   可以救回大部分订单
   
   投入 2 周 vs 节省百万损失，ROI 极高
```

### 故事 4：DB 慢查询治理（救火）

**Situation**：

```
某周二下午:
  - 用户反馈下单慢
  - 监控告警 P99 飙到 5s
  - 错误率 30%
```

**Task**：

```
紧急止血 + 根因 + 长期防御
```

**Action**：

```
T+0 5min: 战时指挥成立
  钉钉拉群（订单 TL + DBA + 运维）
  开实时监控大盘
  评估影响范围

T+0 8min: 一级降级
  关闭推荐 / 评论
  减少非核心 DB 访问

T+0 10min: 限流
  网关限流到 5k QPS
  保护 DB

T+0 15min: 部分恢复

T+0 20min: 定位
  SHOW PROCESSLIST 看到大量 SELECT 卡住
  EXPLAIN: type=ALL，全表扫
  SQL: SELECT * FROM orders WHERE user_id IN (...大量 ID...)
  
  根因: 营销服务上线新功能
        没用索引 + IN 列表 5000 个

T+0 30min: 临时修复
  KILL 慢查询
  通知营销下线该功能
  DB 恢复

T+0 45min: 完全恢复

事后根因 5 Whys:
  Why 1: DB 慢
  → 慢查询打满 CPU

  Why 2: 慢查询从哪来
  → 营销服务新接口

  Why 3: 为什么没发现
  → 没走 SQL Review

  Why 4: 为什么没 SQL Review
  → 流程缺失

  Why 5: 为什么流程缺失
  → 团队大了，新人不知道

长期改进:
  1. SQL Review 流程化
     - CI 集成 sqlcheck
     - 高危 SQL 工单 + DBA 审核
     - 新服务上线 checklist
  
  2. 慢查询自动告警
     - long_query_time = 1s
     - 告警分级（>1s 警告 / >5s P0）
     - 自动通知开发 + DBA
  
  3. 限流自动触发
     - DB 负载 > 80% 自动限流
     - 不依赖人工
  
  4. 团队培训
     - DB 规范培训
     - Onboarding checklist
     - 季度 DB 故障复盘分享
```

**Result**：

```
本次故障:
  - 影响时长 45 分钟
  - 用户感知失败 5 万次
  - GMV 损失约 100 万

后续改进效果:
  - SQL Review 拦截 200+ 高危 SQL
  - 慢查询告警提前发现 50+ 隐患
  - 类似故障 0 复发
  - DB 稳定性提升 99.9% → 99.99%
```

**Learnings**：

```
1. 反思 - 故障应急的"15 分钟法则":
   止血优于定位
   先恢复服务，再找根因
   团队对此达成共识，后续故障平均恢复时间从 1h → 15min

2. 反思 - 流程比技术更重要:
   这次问题不是技术问题
   是流程问题（没 Review）
   补流程 + 工具自动化 才能根治
   
   教训: 资深工程师要推动流程，不只是写代码

3. 反思 - 战时指挥的价值:
   钉钉群 + 实时大盘 + 角色分工
   让响应从 30min → 5min
   后来固化为 Runbook
```

### 故事 5：从单体到微服务

**Situation**：

```
2023 年初:
  - 单体 Java 应用（接手时是 Java，后改 Go）
  - 50w 行代码
  - 一个 jar 50 个模块
  - 改 1 行代码全部重启
  - 部署 30 分钟
  - 上线一次 1 小时
  - 团队扩到 30+ 人，沟通成本爆炸
```

**Task**：

```
6 个月内:
  - 拆 5-7 个微服务
  - 不影响业务迭代
  - 团队效率提升 50%+
```

**Action**：

```
1. DDD 战略设计 (2 周)
   事件风暴 → 识别限界上下文
   按业务拆分:
     订单上下文 (Order)
     商品上下文 (Product)
     库存上下文 (Stock)
     支付上下文 (Payment)
     营销上下文 (Marketing)
     物流上下文 (Logistics)
     用户上下文 (User)

2. 渐进式拆分（4 个月）
   策略: 绞杀者模式
   - 新功能放新服务
   - 老功能逐步搬迁
   - 双写过渡
   - 切流 + 老代码下线

3. 技术栈选型
   - 框架: Kratos（B 站开源，gRPC 主导）
   - 注册中心: Nacos
   - 配置中心: Nacos
   - 网关: APISIX
   - 服务通信: gRPC
   - MQ: Kafka
   - 链路追踪: Jaeger + OpenTelemetry
   - 监控: Prometheus + Grafana
   - 日志: Loki

4. 拆分关键决策
   - 数据所有权: 每个服务独立 DB
   - 服务通信: 强一致 → gRPC，最终一致 → MQ
   - 跨服务事务: TCC + Saga + Outbox
   - 跨服务查询: BFF 聚合 / CQRS

5. 配套建设
   - 服务模板（避免重复造）
   - CI/CD pipeline
   - 灰度发布
   - 全链路压测
   - 故障演练（混沌工程）
```

**Result**：

```
量化结果:
  - 部署时间 30min → 3min
  - 单服务规模 < 3w 行
  - 团队解耦（小团队负责小服务）
  - 业务迭代速度 2-3x
  - 故障爆炸半径减小（隔离）

长期价值:
  - 后续 1 年新业务全用微服务
  - 推动了公司的微服务标准化
  - 团队人才梯队化（架构 / 业务 / 运维）
```

**Learnings**：

```
1. 反思 - 拆得过细的痛:
   一开始想拆 15 个服务
   后来发现 5-7 个最合适
   过细 → 分布式泥潭（事务 / 调用链 / 运维）
   
   教训: 拆分粒度按团队规模和业务复杂度
   不是越细越好

2. 反思 - 数据迁移最痛:
   双写 + 校验花了大部分时间
   特别是历史数据格式不规范
   
   教训: 单体时就要重视数据规范

3. 反思 - DDD 落地有阻力:
   团队习惯贫血模型
   推 DDD 阻力大
   后来选择"轻 DDD"折中
   
   教训: 方法论要因地制宜
   不能全套照搬
```

### 故事 6：缓存一致性优化

**Situation**：

```
现象:
  - 用户更新地址
  - 部分实例缓存仍是旧地址
  - 物流发到旧地址
  - 客诉激增
```

**Task**：

```
解决多实例缓存一致性
  + 不影响性能
  + 业务无感知
```

**Action**：

```
方案对比:

方案 A: 直接读 DB（强一致）
  ✗ 性能差，DB 撑不住

方案 B: Cache-Aside + 双删
  ✓ 减少不一致窗口
  ✗ 仍非强一致

方案 C: binlog 订阅 + 删缓存（最终选）
  ✓ 解耦
  ✓ 删除可靠
  ✗ 延迟 100ms-1s

方案 D: 多级缓存 + Pub/Sub 失效
  本地 + Redis + DB
  本地 TTL 短（5s）
  Redis TTL 中（30min）

最终: B + C + D 组合

读路径:
  本地 → Redis → DB → 回填

写路径:
  写 DB → 删 Redis（异步重试）
  binlog → Canal → Kafka → 删 Redis（兜底）
  Kafka 广播 → 各实例删本地缓存

实施细节:
  本地缓存: Ristretto
    Capacity = 100MB
    TTL = 5s（短，接受短期不一致）
    LFU 淘汰

  Redis:
    Cluster 32 节点
    TTL = 30min + 随机 5min（防雪崩）
    Lua 脚本删除（原子）

  binlog 订阅:
    Canal → Kafka topic: order_change
    消费端: 删 Redis + 广播失效本地

  超热点（爆款商品）:
    本地缓存 30s（接受不一致）
    异步刷新

  关键指标埋点:
    - 缓存命中率
    - 不一致窗口（写后多久缓存还是旧）
    - 删除失败率
```

**Result**：

```
量化结果:
  - 缓存命中率: 95%+
  - 不一致窗口: P99 < 200ms
  - DB 压力: 减少 90%
  - 客诉: 减少 95%

复用价值:
  - 多级缓存组件抽出来
  - 全公司 5+ 业务复用
```

**Learnings**：

```
1. 反思 - 一致性 vs 性能取舍:
   没有完美方案，都是权衡
   接受短期不一致换性能

2. 反思 - 监控比方案更重要:
   不一致窗口 / 命中率 / 删除失败
   这些指标让我们能持续优化

3. 反思 - 抽组件复用:
   一开始只在订单用
   抽成通用 SDK 后全公司用
   
   教训: 资深工程师要看长远，做能复用的东西
```

## 六、Learnings 整体反思（讲项目时用）

```
3 年下来，最大的 5 个学到:

1. 一致性是设计问题，不是补救问题
   一开始想靠对账兜底，后来发现治标不治本
   从根上加幂等 + 状态机才是王道

2. 监控比方案更重要
   方案再好，看不到指标就没法持续优化
   实时大盘 + 多维度埋点 = 资深的标志

3. 流程比技术更重要
   90% 故障是流程问题（无 Review / 无演练 / 无 Runbook）
   补流程 + 工具自动化才能根治
   资深工程师要推动流程

4. 不要追求"完美方案"
   每个方案都是权衡（一致 vs 可用 / 性能 vs 成本）
   关键是说清取舍和后续演进

5. 做能复用的东西
   一次解决问题 → 抽组件 → 全公司复用
   这是从"高级开发"到"架构师"的关键
```

## 七、追问应对预案

### 追问 1: "为什么不用 TiDB"

```
"评估过 TiDB:
  优点: 自动分片 + 兼容 MySQL + HTAP
  缺点: 
    - 团队不熟，运维风险高
    - P99 延迟比 MySQL 略高（10ms vs 5ms）
    - 小事务性能略差
  
  我们当时:
    - 业务规模 16 库分库够 5 年
    - 团队对 MySQL 极熟
    - 风险偏好低（业务在快速增长）
  
  → 选 MySQL 分库分表
  
  现在回头看:
    - 如果数据 > 10TB 或跨地域，会上 TiDB
    - 业务超过分库能力上限再切换"
```

### 追问 2: "Redis 集群挂了怎么办"

```
"分情况:
  
  单分片挂:
    Cluster 自动切换从节点（30s）
    业务影响: 该 slot 30s 不可用
    应对: 多级缓存（本地 5s TTL）兜底

  整集群挂:
    切流到备份集群（DBA 切，1-2 分钟）
    本地缓存继续顶
    熔断 → DB 直读 + 限流（保护 DB）

  根本上:
    业务不能 100% 依赖 Redis
    所有缓存逻辑必须有 fallback
    DB 是 source of truth"
```

### 追问 3: "怎么做大促容量评估"

```
"完整流程:

  T-90: 业务侧确认大促玩法 + 预估流量
    - 历史数据（去年同期 × 增长系数）
    - 营销侧预估（投放规模）
    - 安全余量（× 1.5）

  T-60: 架构侧评估
    - 各服务容量（按 QPS / RT 推算）
    - 中间件容量（DB / Redis / MQ）
    - 网络带宽
    - 输出资源清单

  T-30: 扩容
    - 应用 pod 扩 5x
    - DB 扩从（读多）
    - Redis 加节点
    - Kafka 加分区
    - CDN 带宽采购

  T-14 ~ T-3: 全链路压测（5 轮）
    - 第 1 轮: 单服务压
    - 第 2 轮: 链路压
    - 第 3 轮: 全链路 + 影子表
    - 第 4 轮: 全链路 + 故障注入
    - 第 5 轮: 最终验证

  T-7: 100+ 预案
    - 限流 / 降级 / 熔断 开关
    - 战时指挥分工
    - Runbook

  T-1: 全员演练

  T-0: 战时值守

  T+7: 复盘 + 缩容"
```

### 追问 4: "TCC 怎么用的"

```
"以订单 + 库存为例:

  Try 阶段:
    - 订单服务: 创建订单（status=TRYING）
    - 库存服务: 预扣库存（available -, frozen +）

  Confirm 阶段（业务操作成功）:
    - 订单服务: 确认订单（status=CONFIRMED）
    - 库存服务: 确认扣减（frozen -）

  Cancel 阶段（业务操作失败）:
    - 订单服务: 取消订单（status=CANCELLED）
    - 库存服务: 释放库存（available +, frozen -）

  3 大坑:
    1. 空回滚: Cancel 时 Try 还没到（用事务记录表幂等）
    2. 悬挂: Try 后 Cancel 比 Try 先到（同上）
    3. 幂等: Confirm/Cancel 可能重发（状态机 + 唯一约束）
  
  框架: Seata / 自研
  我们用 Seata + 业务层补偿
  关键路径仍加对账兜底"
```

### 追问 5: "讲一次失败的项目"

```
"印象深的:

  2023 年试图把订单状态机改成事件溯源（ES）
  
  动机:
    - 想做完整审计
    - 想支持回放
    - DDD 推荐
  
  做了 3 个月:
    - 引入 EventStore
    - 重写订单状态变更逻辑
    - 改读模型为投影
  
  失败原因:
    1. 团队不熟 ES → 学习成本高
    2. 调试困难（事件流难追踪）
    3. 性能比传统设计差
    4. 业务收益不明显（审计完全可以用 binlog 替代）
  
  最终:
    - 上线 1 个月发现得不偿失
    - 回滚到传统设计
    - 损失 3 个月研发
  
  反思:
    - 选型不能为技术而技术
    - 要看团队能力 + 业务收益
    - 大改造要先 POC 验证
    - 我作为 TL 应该更早叫停
  
  后来:
    - 推任何新方案前先 POC + 评审
    - 阶段性 Go/No-Go 决策
    - 不被'方法论'绑架"
```

### 追问 6: "怎么和产品 PK"

```
"原则:
  - 不直接说'做不了'
  - 说'要 X 代价 + Y 风险 + Z 时间'
  - 给产品备选方案
  - 数据说话

  例子:
    产品: '希望大促当天上新功能 X'
    
    我: '这个功能开发需要 2 周。
         大促前 1 周冻结代码（避免风险）。
         如果现在开始，能赶上但 QA 时间紧。
         
         3 个备选:
         A. 全量开发 + 灰度上线（推荐）
         B. 简化版本先上 + 大促后完善
         C. 大促后再做（最保险）
         
         你看怎么取舍？'
  
  关键:
    - 把决策权交给产品
    - 但提供专业判断
    - 共同对结果负责"
```
