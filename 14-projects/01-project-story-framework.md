# 项目故事框架

> 好项目不是“我用了很多技术”，而是“我在明确约束下解决了具体问题，并证明结果有效”。

## 一、STAR + 技术取舍

传统 STAR：

| 部分 | 含义 |
| --- | --- |
| Situation | 背景 |
| Task | 目标 |
| Action | 动作 |
| Result | 结果 |

技术面试要加两层：

```text
Tradeoff：为什么这么设计，放弃了什么
Review：上线后暴露什么问题，如何改进
```

完整结构：

```text
背景 -> 目标 -> 约束 -> 方案 -> 取舍 -> 落地 -> 指标 -> 复盘
```

## 二、1 分钟项目介绍

模板：

```text
这个项目是做 {业务目标}，主要解决 {核心问题}。
我负责 {模块/链路}，当时规模大概是 {QPS/数据量/用户量/订单量}。
技术上最关键的是 {难点 1} 和 {难点 2}。
最后我们通过 {方案} 把 {指标} 从 A 优化到 B，线上稳定性也通过 {监控/降级/补偿} 保障。
```

## 三、技术难点展开

按这个顺序讲：

1. 现象：为什么这是问题。
2. 约束：不能简单怎么做。
3. 方案：你做了什么。
4. 取舍：为什么选它。
5. 结果：指标变化。
6. 复盘：还有什么不足。

示例：

```text
当时订单列表查询 P99 很高，主要原因是数据量上来后深分页和多条件查询导致 MySQL 扫描行数过多。
我们没有直接上复杂搜索系统，而是先按查询模式重新设计联合索引，并把历史订单做冷热拆分。
对于复杂筛选，后续同步到 ES 承接。
这样做的好处是改动小、风险低，缺点是存在数据同步延迟，需要用状态修复和对账兜底。
上线后 P99 从 800ms 降到 120ms 左右。
```

## 四、结果怎么量化

不要只说“性能提升明显”。

可以说：

- P99 从 800ms 降到 120ms。
- 慢 SQL 数量下降 80%。
- DB CPU 从 80% 降到 40%。
- 消息积压从 300 万降到 0。
- 人工处理时间从 2 小时降到 10 分钟。
- 故障恢复时间从 30 分钟降到 5 分钟。

## 五、常见错误表达

- “这个项目并发很高”但没有量级。
- “用了 Redis、MQ、ES”但说不清为什么。
- “解决了性能问题”但没有指标。
- “我们做了高可用”但没有故障场景。
- “我参与了”但说不清自己负责什么。

## 六、面试表达

```text
我讲项目会先控制在 1 分钟，让面试官知道业务、规模、我的职责和两个技术难点。
如果继续追问，我会展开其中一个难点，重点讲约束、方案对比、架构取舍和结果指标。
我不会只罗列技术栈，因为项目价值在于解决了什么问题，以及上线后是否稳定。
```
