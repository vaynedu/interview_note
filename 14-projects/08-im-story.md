### 故事 1：500w 长连接架构

**Situation**：

```
2023 接手时:
  - 单机 5w 长连接（Java + Netty）
  - 在线 100w → 20 台机器
  - 重连风暴一波打挂 5 台
  - 内存占用 32GB（连接对象太重）
```

**Task**：

```
6 个月内:
  - 单机支持 30w 长连接（5x）
  - 撑住 500w 在线（实际用 20 台）
  - 重连有限流保护
  - 内存控制在 16GB 内
```

**Action**：

```
方案对比:

A. 继续优化 Java + Netty
   ✓ 团队熟
   ✗ JVM GC 影响延迟
   ✗ 内存效率低

B. Go + gnet / netpoll
   ✓ goroutine 轻量（连接对象小）
   ✗ 团队学习成本

C. C++ 自研
   ✗ 开发成本高
   ✗ 团队不熟
   → 否

最终选 B（Go + gnet）

技术细节:

1. 网络模型
   gnet (epoll based) vs net.Conn (goroutine-per-conn)
   gnet 单 epoll loop，零拷贝
   单机 30w 连接 内存 8GB

2. 连接对象优化
   旧: 每连接 50KB（Java 对象 + 缓冲区）
   新: 每连接 5KB（Go 紧凑结构）
   
   关键设计:
   - 用对象池复用 buffer
   - 紧凑 struct (无业务字段嵌入)
   - lazy 业务字段（按需分配）

3. 心跳设计
   客户端: 每 30s 发心跳
   服务端: 60s 无心跳判断断
   
   优化:
   - 心跳合并（10s 窗口收集）
   - 心跳协议精简（4 byte）
   - 服务端用时间轮检测超时

4. 重连风暴防护
   客户端:
   - 指数退避（1s → 2 → 4 → 8 → 30s 上限）
   - 随机 jitter（避免同时重连）
   
   服务端:
   - 限流: 单 IP 每秒 10 连接
   - 总量限流: 全局 1w 连接/秒
   - 验签放后置（先建连，慢慢校验）

5. 多机路由
   接入层无状态:
   - 任意机器都能接受连接
   - 用户 ID → 路由表（Redis）
   - 心跳 + Watch 维护路由
   
   消息发送:
   - 查路由表 → 找到目标用户在哪台机器
   - 直接 RPC 到对应机器

6. 优雅下线
   滚动升级:
   - 流量摘除（注册中心下线）
   - 新连接路由到其他机器
   - 老连接自然结束（30 分钟 grace period）
   - 强制断开剩余（客户端会重连）
```

**Result**：

```
量化结果:
  - 单机长连接: 5w → 30w (6x)
  - 内存/连接: 50KB → 5KB (10x)
  - 接入机器: 100 台 → 20 台
  - 端到端 P99: 1s → 80ms
  - 重连风暴 0 影响

业务价值:
  - 服务器成本节省 80%
  - 用户体验显著提升
  - 撑住业务增长 3x（从 100w → 500w 在线）
```

**Learnings**：

```
1. 反思 - 选型要看长远:
   一开始有人提"先优化 Java"
   评估发现极限就是 10w/机器
   一年后还得换 Go
   不如直接换
   
   教训: 技术债早还早轻松

2. 反思 - 客户端要配合:
   服务端做了很多优化
   但客户端不配合（如不退避）
   也撑不住
   
   教训: SDK 设计也要重视
   推动客户端团队改造比服务端难

3. 团队收获:
   团队从 Java 转 Go
   建立了 Go 技术栈
   后续业务都用 Go
```

### 故事 2：推拉结合的消息分发

**Situation**：

```
原架构（推模式）:
  发送方 → 服务端 → 推送给所有接收方
  
  问题:
  - 大群（1w 人）一条消息扩散 1w 次
  - 离线用户也推（浪费）
  - 多端同步靠拉
```

**Task**：

```
设计推拉结合方案
  + 大群消息高效
  + 多端同步一致
  + 离线消息可靠
  + 实时性 < 500ms
```

**Action**：

```
设计原则:
  - 在线推（长连接，实时）
  - 离线拉（连上时同步）
  - 大群拉（避免扩散）
  - 小群推（实时性好）

数据模型:

1. 消息存储
   message 表:
     msg_id (雪花算法，趋势递增)
     conversation_id (会话 ID)
     sender_id
     content (JSON)
     created_at
     msg_type (text / image / ...)
   
   分库分表:
   - conversation_id Hash 分 64 库
   - 时间分表（月表）

2. 会话状态
   conversation_state:
     conversation_id
     user_id
     last_read_msg_id  -- 已读位置
     last_sync_msg_id  -- 同步位置
     mute (静默)
     pinned (置顶)

3. 用户信箱（推模式）
   user_inbox (Redis ZSet):
     KEY: user_inbox:{user_id}
     SCORE: msg_id (趋势递增)
     MEMBER: msg_id

   只放 conversation_id（不放消息内容）
   保留最近 1000 条

分发逻辑:

1. 单聊 / 小群（< 200 人）— 推模式
   发送方 → 消息持久化 → 写入接收方 user_inbox
   → 长连接推送（如果在线）
   → 推送通道（如果离线）

2. 大群（>= 200 人）— 拉模式
   发送方 → 消息持久化（只一份）
   → 在线用户长连接推送（仅通知"有新消息"）
   → 客户端拉取消息（按 conversation_id + 增量）

3. 离线消息同步
   客户端上线 → 携带 last_sync_msg_id
   服务端 → 拉取 > last_sync_msg_id 的所有消息
   分批返回（避免单次过大）
   客户端确认 → 更新 last_sync_msg_id

4. 多端同步
   用户 ID 维度，不区分设备
   每个设备有自己的 last_sync_msg_id
   消息存一份，N 个设备各自拉自己的

技术细节:

1. 消息 ID 设计
   雪花算法 + 趋势递增
   优势:
   - 排序方便
   - 客户端去重容易
   - 增量拉取（> last_msg_id）

2. 大群优化
   - 不写每个用户 inbox（爆炸）
   - 只写群 inbox（一份）
   - 读时按权重分发推送优先级
     - 在线用户立刻推
     - 活跃用户优先推
     - 不活跃用户延迟推 / 合并推

3. 推送降级
   - 长连接 在线推
   - APNS / FCM 兜底
   - 厂商通道（小米 / 华为 / OPPO / vivo）
   - SMS 极端兜底（关键消息）

4. 顺序保证
   - 同会话同分区（Kafka）
   - 单线程消费
   - 消息 ID 严格递增
```

**Result**：

```
量化结果:
  - 大群（10w 人）消息扩散: 10w 次写 → 1 次写 + N 次读
  - 存储成本降 80%
  - 端到端延迟: 1s → 200ms
  - 离线消息可靠性: 95% → 99.99%
  - 推送到达率: 70% → 95%（多通道兜底）
```

**Learnings**：

```
1. 反思 - 推 / 拉混合是 IM 标准:
   纯推（写扩散）小群 OK，大群崩
   纯拉（读扩散）实时性差
   混合是最优解
   
   教训: 没有银弹，要因场景选

2. 反思 - 推送渠道一定要多:
   单一通道（如只 APNS）极不稳定
   厂商通道 + APNS + 长连接 多管齐下
   
   教训: 推送是 IM 命脉，不能省

3. 反思 - 客户端体验关键:
   服务端再快，客户端展示慢也白搭
   推动客户端做本地数据库 + 后台预拉
   
   教训: 端到端体验 = 服务端 + 网络 + 客户端
```

### 故事 3：30w 人大群消息

**Situation**：

```
业务上线"全员群"功能
  - 公司全员 5w 人在一个群
  - 老板发消息 → 全员收到
  - 高峰每秒 100 条消息
  - 现有架构挂了
```

**Task**：

```
1 个月内:
  - 支持 10w 人大群
  - 消息延迟 < 1s
  - 不影响其他业务
```

**Action**：

```
分析:
  - 写扩散: 1 条消息 → 10w 次推送 → 死
  - 读扩散: 全员同时拉 → 服务端压力大
  - 必须组合 + 限流

设计:

1. 写存储优化
   1 条消息只存 1 份（群 inbox）
   不写 10w 个用户 inbox
   存储压力 / 10w

2. 推送优化（关键）
   分批推送 + 优先级:
     - P0: 在线 + 活跃 → 100ms 内推
     - P1: 在线 + 不活跃 → 1s 内推
     - P2: 离线 → 慢推 / 合并推
   
   推送队列分级:
     - 高优先级队列（直接推长连接）
     - 中优先级（10s 批量推）
     - 低优先级（1min 批量推 + APNS 合并）

3. 客户端拉取
   长连接收到"新消息通知"
   增量拉取（不是全量）
   按 ON_VISIBLE 时拉（用户打开会话才拉）
   避免后台静默用户拉

4. 限流降级
   单群消息频率限流（每秒 10 条）
   超过自动合并（"今天有 100 条新消息"）
   
   极端情况降级:
   - 关闭 @所有人 推送
   - 关闭振动 / 提示音
   - 只推 5% 用户测试

5. 路由优化
   群消息发到 Kafka topic: group_msg
   按 group_id Hash 分区
   消费端按权重分发推送任务
```

**Result**：

```
量化结果:
  - 支持单群 10w 人（实际用 5w）
  - 端到端延迟 P99 < 800ms
  - 服务端 CPU 高峰 30%
  - 不影响其他业务

业务价值:
  - 全员群上线后老板满意
  - 后续支持各种"行业群" / "活动群"
```

**Learnings**：

```
1. 反思 - 限流不是消极策略:
   一开始不愿做限流（怕用户体验差）
   后来发现:
   - 没限流 → 全挂
   - 有限流 + 合并 → 优雅降级
   
   教训: 限流是保护用户，不是限制用户

2. 反思 - 推送优先级很关键:
   不是所有用户都同等重要
   活跃用户优先 / 离线用户合并
   服务端负载 / 用户体验都好
   
   教训: 推送策略 = 体验设计

3. 反思 - 客户端配合不可少:
   ON_VISIBLE 拉取（用户看才拉）
   静默用户不浪费资源
   
   教训: 端 + 云协同，光优化云没用
```

### 故事 4：大规模断连风暴（救火）

**Situation**：

```
某天晚上 8 点（高峰）:
  - 监控告警: 长连接数从 400w 骤降到 50w
  - 客服群爆炸
  - 老板拉群
```

**Task**：

```
紧急止血 + 根因 + 长期防御
```

**Action**：

```
T+0 5min: 战时指挥
  拉所有 owner（IM 网关 / 网络 / 推送 / 客户端）
  开监控大盘
  评估范围

T+0 10min: 止血
  - 路由层强制限流（每秒新连接 5w 上限）
  - 防止重连风暴打挂剩余机器
  - 关闭非核心推送（只保留核心消息）

T+0 15min: 定位
  - 网络组确认: 上海机房光缆故障
  - 客户端日志: 大量"连接超时"
  - 服务端日志: 长连接被强制断开

T+0 30min: 部分恢复
  - 备用线路接入
  - 客户端开始重连成功

T+0 1h: 完全恢复
  - 长连接数恢复到 380w
  - 还有 20w 客户端有问题（找不到回原机器）

T+0 2h: 全面恢复
  - 所有客户端重连
  - 监控所有指标正常

事后根因 5 Whys:
  Why 1: 用户掉线
  → 长连接被网关强制断

  Why 2: 网关为什么断
  → 上海机房光缆故障，网络抖动 30s

  Why 3: 30s 抖动为什么影响大
  → 客户端心跳设 60s，30s 后必然断

  Why 4: 重连为什么花了 2h
  → 重连风暴 + 限流导致排队

  Why 5: 为什么没演练
  → 没做过大规模断连演练

长期改进:
  1. 客户端心跳从 60s → 30s
     更快感知断连
     避免"假连接"

  2. 多机房接入
     上海 + 北京 + 深圳
     单机房挂只影响 1/3 用户
     DNS 智能解析

  3. 重连优化
     - 指数退避更激进（1s → 30s 上限）
     - 客户端记忆"最近成功的接入点"
     - 服务端"快速通道"（VIP / 老用户优先）

  4. 重连风暴演练
     每月演练一次
     模拟 50% 用户同时重连
     验证限流 + 路由

  5. 监控加强
     - 长连接数 5 分钟环比降 10% 告警
     - 单机房连接数偏离告警
     - 重连 QPS 告警
```

**Result**：

```
本次故障:
  - 影响 350w 用户
  - 平均不可用 30 分钟
  - 客服投诉 5w+

后续效果:
  - 多机房上线后，类似故障影响 < 1/3 用户
  - 重连演练发现 3 个 bug 提前修复
  - 监控提前 5 分钟发现潜在问题
  - 一年内 0 大规模断连事故
```

**Learnings**：

```
1. 反思 - 演练是真演练:
   平时说"我们有多机房"
   真出事才发现根本没切过流
   
   教训: 没演练过的预案 = 不存在

2. 反思 - 客户端心跳的取舍:
   60s 心跳省服务端
   30s 心跳更敏感
   选 30s 损失服务器，但用户体验好
   
   教训: 不要为了省一点资源牺牲核心体验

3. 反思 - 复盘的价值:
   这次事故催生 5 个改进
   1 年内类似问题 0 复发
   
   教训: 故障是"礼物"，复盘是"利息"
```

### 故事 5：跨机房消息同步

**Situation**：

```
业务上线"国际版"
  - 海外用户访问国内机房延迟 200-500ms
  - 国内用户体验也变差（共用资源）
  - 监管要求海外数据本地化
```

**Task**：

```
3 个月内:
  - 国内机房 + 海外机房
  - 跨机房消息可达
  - 国内 / 海外用户互发消息 < 1s
  - 数据合规分离
```

**Action**：

```
方案设计:

1. 路由层
   DNS 智能解析:
   - 国内用户 → 国内接入点
   - 海外用户 → 海外接入点
   - 各自连本地长连接网关

2. 消息存储分离
   - 国内消息存国内（用户 + 数据）
   - 海外消息存海外
   - 跨机房消息双写（合规允许的部分）

3. 跨机房同步
   消息复制方案:
   - Kafka MirrorMaker 跨机房
   - 国内 → 海外 topic 同步
   - 海外 → 国内 topic 同步
   - 单向 / 双向按业务定

4. 用户路由
   接入层判断目标用户在哪个机房:
   - 同机房 → 直接推
   - 跨机房 → RPC 到对方机房

5. 弱网优化（海外）
   - 客户端协议精简（protobuf）
   - 消息压缩（zstd）
   - 长连接 + QUIC 协议

6. 数据合规
   - 用户数据按属地存储
   - 跨境消息明确同意
   - 加密传输 + 加密存储
```

**Result**：

```
量化结果:
  - 海外用户延迟: 500ms → 100ms
  - 国内用户体验恢复
  - 跨机房消息延迟: < 1s
  - 数据合规通过审计

业务价值:
  - 国际化业务上线
  - 海外用户增长 5x
```

**Learnings**：

```
1. 反思 - 跨地域是巨坑:
   网络延迟 / 数据合规 / 时区 / 编码
   每一个都是大坑
   
   教训: 跨地域要预留 2x 时间

2. 反思 - 数据合规越来越严:
   GDPR / 各国数据法律
   早期不重视 → 后期补成本极高
   
   教训: 涉及国际化要早期合规

3. 反思 - QUIC 协议价值:
   弱网下 QUIC 比 TCP 好很多
   建链快 + 多流复用 + 0-RTT
   
   教训: 协议层优化很重要
```

### 故事 6：消息已读 / 未读

**Situation**：

```
业务上线"已读未读"功能
  - 单聊已读容易
  - 群聊已读复杂（10w 人群每个人都标已读？）
  - 多端同步（手机已读电脑也要变）
```

**Task**：

```
2 个月内:
  - 单聊已读未读
  - 群聊已读人数（不实时显示每个人）
  - 多端同步
  - 性能不影响其他功能
```

**Action**：

```
数据模型:

read_state:
  conversation_id
  user_id
  last_read_msg_id  -- 已读到哪条消息
  read_count        -- 群聊用，已读人数

设计:

1. 单聊已读
   用户阅读消息 → 上报 last_read_msg_id
   服务端更新 read_state
   通知对方更新 UI（长连接推）

2. 群聊已读
   不存"每个人是否已读"（爆炸）
   只存 read_count（已读人数）
   
   流程:
   - 用户阅读 → 上报 last_read_msg_id
   - 服务端原子 INCR read_count（仅当 last_read_msg_id 未达到时）
   - 群消息列表显示"X / Y 人已读"

3. 多端同步
   用户 A 在手机读了 → 上报
   服务端记录 user_id 维度的 last_read_msg_id
   推送给该用户其他设备
   电脑端收到 → 本地标记已读

4. 性能优化
   - 已读上报合并（10s 窗口）
   - 不阻塞主链路（异步）
   - read_state 用 Redis Hash（user_id → state）
   - 定期持久化到 DB

5. 已读隐私
   产品支持"关闭已读回执"
   一方关闭 → 双方都看不到
```

**Result**：

```
量化结果:
  - 单聊已读延迟: < 200ms
  - 群聊已读统计准确率: 99.9%
  - 多端同步延迟: < 500ms
  - 不影响主链路性能

业务价值:
  - 已读功能上线后用户活跃度 +15%
  - 用户反馈正面
```

**Learnings**：

```
1. 反思 - 群已读不能存所有人:
   10w 人群存 10w 条已读 = 爆炸
   只存 count 即可（产品同意不显示具体谁读）
   
   教训: 数据规模决定设计
   有些"完美"功能要妥协

2. 反思 - 隐私设计要早期:
   "已读回执"是隐私敏感功能
   一开始没想，上线后用户投诉
   后续加"关闭"开关
   
   教训: 隐私是产品的核心，不是事后补救
```

## 七、追问应对预案

### 追问 1: "为什么不用 WebSocket"

```
"我们也用 WebSocket，但有补充:
  - 弱网下 WebSocket 重连慢
  - 移动端切网（WiFi → 4G）必断
  - 客户端实现兼容性差

  我们的栈:
  - 主流: WebSocket
  - 弱网 / 海外: QUIC
  - 移动端: 自研协议（基于 TCP，更轻量）
  
  根据用户网络环境自动协商最佳协议"
```

### 追问 2: "怎么保证消息不丢"

```
"3 层保证:

  L1 客户端 ACK:
    收到消息 → 客户端发 ACK
    服务端收到 ACK → 标记已送达
    没 ACK → 重发 3 次

  L2 服务端持久化:
    消息先持久化 DB
    再分发推送
    持久化失败 → 拒绝接收（让发送方重试）

  L3 离线兜底:
    消息存 user_inbox
    用户上线 → 增量拉取
    last_sync_msg_id 保证一致

  + 监控:
  - 消息送达率
  - ACK 失败率
  - 离线拉取数量
  
  现在做到 99.99% 不丢"
```

### 追问 3: "怎么保证消息不重"

```
"客户端去重:
  - 消息 ID 全局唯一（雪花算法）
  - 客户端本地维护已收消息 ID 集合
  - 重复 ID 直接丢弃

服务端去重:
  - 发送时分配 message_id（一次性）
  - 数据库唯一索引保证不重复存
  - 推送有重试，但客户端去重兜底"
```

### 追问 4: "怎么保证消息顺序"

```
"3 层保证:

  L1 消息 ID 趋势递增:
    雪花算法保证
    客户端按 ID 排序

  L2 同会话同分区:
    Kafka 按 conversation_id Hash 分区
    单分区单消费者保顺序

  L3 客户端容错:
    收到乱序 → 等待 / 排序
    超时 → 触发拉取（修复）

  极端场景（多端同时发）:
    服务端时间戳 + 客户端时间戳
    服务端时间戳为准
    冲突时 sender_id 决胜"
```

### 追问 5: "10w 人大群怎么做"

```
按故事 3 完整答。要点:
  - 1 条消息只存 1 份（群 inbox）
  - 不写每个用户 inbox
  - 推送分优先级（在线 + 活跃优先）
  - 客户端 ON_VISIBLE 拉
  - 限流 + 合并
```

### 追问 6: "海外用户体验怎么保"

```
按故事 5 完整答。要点:
  - 多机房（国内 + 海外）
  - DNS 智能解析
  - QUIC 协议
  - 数据本地化（合规）
  - 跨机房同步用 Kafka MirrorMaker
```
