# 团队管理实战案例

> Leadership 答题不能只讲方法论，要有**真实案例**。本篇 15 个**可直接复用的故事**：招聘 / 1v1 / 绩效 / 冲突 / 离职 / 跨部门 / 救火 / 推动改进。
>
> 与 [04-senior-answers.md](04-senior-answers.md) 配合使用。
>
> ⚠️ 不要原样背 —— 替换团队规模 / 业务 / 数字，转成你自己的故事。

---

## 一、为什么要有"实战案例"

```
面试官追问:
  "你怎么招人？"
  → 你答框架 ✓ → 接着问 "讲个例子"
  → 答不上具体案例 → 给人"光说不练"印象

资深面试 = 框架 + 案例
  框架体现思考
  案例体现真做过
  缺一不可
```

## 二、案例 1：招到一个改变团队的人

### Situation

```
背景:
  我团队 8 人做电商订单，2024 Q2
  缺一个 senior 后端
  招了 3 个月，面试 30+ 简历，没合适的
  
  问题:
  - HR 简历筛过来不准
  - 团队"看脸"招人
  - bar 不一致
```

### Task

```
1 个月内招到合适 senior:
  □ 8 年+ Go 经验
  □ 高并发系统经验
  □ 能带 1-2 个 junior
```

### Action

```
我的改进:

1. 重新定义岗位 (Job Posting)
   - 之前: 笼统"高级 Go 工程师"
   - 之后: 具体 "订单系统 owner，需要 X / Y / Z 经验"
   - HR 筛选准确率提升

2. 改进面试流程
   - 之前: 4 轮各自为战
   - 之后:
     1 轮: 简历 + HR 筛 (基础匹配)
     2 轮: 1h 技术 + 项目 (我亲自，深度技术)
     3 轮: 1h 系统设计 (架构师参与)
     4 轮: 1h 行为 + 文化 (老板 + HR)
     5 轮: 30min 团队互动 (junior 体验)

3. 标准化评分
   - 5 维度评分: 技术 / 设计 / 业务 / 协作 / 文化
   - 每维度 1-5 分
   - 综合 > 20 推 offer

4. 主动招聘 (不等简历来)
   - LinkedIn / GitHub 主动 reach
   - 内推奖励双倍
   - 自己出去做技术分享 (技术品牌)

5. Offer 谈判
   - 给到行业 P75 (中上)
   - 期权 / 个人发展机会
   - 不为省钱失去人才
```

### Result

```
1 个月内招到张三:
  - 8 年 Go + 字节出身
  - 高并发系统 owner 经验
  - 文化匹配度高

入职后:
  - 1 个月内主导一次架构 review
  - 3 个月内 Mentoring 2 个 junior
  - 6 个月晋升 P7

后续:
  - 我的招聘流程被推到全公司
  - 团队招聘效率提升 3 倍
  - 候选人体验好 (offer 接受率 80%+)
```

### Learnings

```
1. 反思 - JD 决定一半:
   笼统 JD 引来不匹配候选人
   具体 JD 让 HR + 候选人都精准

2. 反思 - 面试官培训:
   每个面试官面试质量参差
   培训 + 评分标准化才公平

3. 反思 - 不要为省钱失去人才:
   省 20% 薪资 → 错过关键人物 → 团队多走 1 年弯路
   ROI 完全不划算

4. 反思 - 招聘是长期投资:
   不是缺人才招
   而是持续看好苗子
   建立人才漏斗
```

### 面试用法

```
被问 "怎么招人 / 招过谁印象深"
→ 用这个案例
→ 重点: 流程化 + 主动 + 标准化
→ 量化: 30 简历 / 1 个月 / 80% 接受率
```

## 三、案例 2：1v1 救回一个想离职的人

### Situation

```
背景:
  团队 senior 李四
  突然提离职 (邮件)
  原因: "想休息"

  我的判断:
  - 李四 6 年员工，技术骨干
  - 最近 3 个月看起来确实低落
  - "想休息"通常不是真原因
```

### Task

```
1 周内:
  - 找出真原因
  - 评估能否挽留
  - 不能挽留也要好聚好散
```

### Action

```
T+0 当天: 1v1 (2 小时)
  - 不直接谈离职
  - 先聊近况 / 家庭 / 兴趣
  - 慢慢引到工作

T+0 暴露真原因:
  - 觉得在团队天花板已到
  - 业务方向不喜欢
  - 老板 (我) 给压力大

T+1 反思:
  - 我确实给压力大 (大促前每天加班催进度)
  - 没有给他成长空间 (一直做老业务)
  - 没问他的兴趣

T+2 行动:
  - 申请新方向给他 (云原生 / DevOps)
  - 减压 (调整他的项目)
  - 安排他参加架构师培训

T+3 二次 1v1:
  - 道歉 + 改进承诺
  - 给具体方案 (新方向 + 减压 + 成长)
  - 让他考虑 1 周

T+10 决定:
  - 李四同意留下
  - 试 3 个月看
```

### Result

```
6 个月后:
  - 李四在新方向做出成绩
  - 主导云原生改造
  - 半年后晋升 P7
  - 现在是团队核心

复盘:
  - 救回 1 个 senior 替代成本 (招聘 + 上手) 至少 50w
  - 团队稳定性提升
  - 我自己的管理方式改进
```

### Learnings

```
1. 反思 - 离职信号不是终点:
   提离职 80% 是有真问题
   不要直接放走
   先了解真原因

2. 反思 - 1v1 不能流于形式:
   平时 1v1 应该听到这些信号
   是我没认真听
   后续 1v1 调整方法 (多问感受)

3. 反思 - leader 是要给空间不是给压力:
   senior 不需要催
   需要的是方向 + 自由 + 成长
   我之前微管理了
```

### 面试用法

```
被问 "讲一次留人经历 / 1v1 经验"
→ 用这个案例
→ 重点: 主动 + 共情 + 反思自己
→ 量化: 6 年员工 / 50w 替代成本 / 6 个月成长 P7
```

## 四、案例 3：辞退一个 senior 但赢得团队尊重

### Situation

```
背景:
  团队 senior 王五
  能力强但合作差:
  - 跟产品吵架 (常上热搜)
  - 看不起 junior (公开嘲讽)
  - 我 1v1 多次提醒无效
  
  团队反映:
  - 不想跟王五合作
  - 影响其他人士气
  - 但王五技术真的牛
```

### Task

```
2 个月内:
  - 改变行为 → 留下
  - 不改变 → 辞退
  - 不影响团队稳定
```

### Action

```
T+0: 正式 1v1 (严肃)
  - 列出具体问题 (3 件事)
  - 听他的解释
  - 给改进期 1 个月
  - 文档化 (PIP 类似)

T+30: 中期 review
  - 没改进 (态度依然)
  - 跟产品又吵了 1 次
  - 给最后机会 1 个月

T+60: 最终决定
  - 仍未改进
  - 决定辞退
  - 跟 HR + 老板对齐

T+60 谈话 (1 小时):
  - 坦诚指出问题
  - 给体面 (协商解除 N+1)
  - 协助找下家 (推荐信)
  - 不情绪化

T+70: 团队沟通
  - 告知团队"经过沟通王五选择新挑战"
  - 不八卦不诋毁
  - 招新替代

T+90: 跟其他团队复盘
  - 分享辞退经验
  - 推动公司"行为不合格"机制
```

### Result

```
团队反应:
  - 90% 表示支持
  - 团队氛围明显好转
  - junior 敢提想法了
  - 跨部门协作改善

王五:
  - 去了另一家公司
  - 1 年后跟我加微信
  - 说我对他帮助大
  - 后来我们还合作过

我的成长:
  - 学会"难的决策要敢做"
  - 团队对我信任增强
  - 后续招聘更看重文化
```

### Learnings

```
1. 反思 - 不能因技术好就容忍:
   一个负面行为 → 影响整个团队
   "好的人才 + 烂的态度" = 净负贡献

2. 反思 - 辞退要有流程:
   PIP + 文档 + 多次沟通
   不能突然
   合规 + 体面

3. 反思 - 团队需要公平:
   有人不合格不处理 = 不公平
   其他人会失去信任
   leader 必须敢做难决策
```

### 面试用法

```
被问 "讲一次辞退 / 难决策"
→ 用这个案例
→ 重点: 流程 + 体面 + 不逃避
→ 量化: 3 件事记录 / 60 天流程 / 90% 团队支持
```

## 五、案例 4：化解技术方案冲突

### Situation

```
背景:
  团队 2 个 senior 争技术方案
  - 张三: 用 Kafka (吞吐高 / 成熟)
  - 李四: 用 Pulsar (新 / 跨地域)
  
  分歧 2 周
  钉钉群吵到老板
  影响项目进度
```

### Task

```
1 周内:
  - 解决技术分歧
  - 维护两人关系
  - 不影响其他成员士气
```

### Action

```
T+0: 单独 1v1 双方
  - 听各自考虑
  - 找出真分歧 (不止技术，还有面子)
  - 让他们冷静

T+1: 重新 review 需求
  - 业务真实需求是什么 (吞吐 100w / 跨地域 否)
  - 团队能力 (Kafka 熟 / Pulsar 不熟)
  - 维护成本

T+2: 组织技术评审
  - 邀请 3 个外部架构师
  - 双方各 30 分钟讲方案
  - 客观打分

T+3: 决策
  - Kafka 胜出 (业务 + 团队 综合)
  - 但保留 Pulsar 部分思路 (跨地域接口设计)

T+4: 收尾
  - 找李四 1v1
  - 肯定他的思考 (跨地域确实是未来方向)
  - 给他主导 Kafka 落地 (展示能力)

T+5: 团队总结
  - 复盘流程 (技术评审标准化)
  - 不再让分歧拖到 2 周
```

### Result

```
项目:
  - 方案确定 1 周后开始
  - 进度按时
  - 上线效果好

人:
  - 张三李四关系恢复
  - 双方都成长 (站对立面看得更全)
  - 团队对"评审制度"信任

机制:
  - 后续技术分歧标准流程:
    1. 单独 1v1 听
    2. 找共同目标
    3. 组织评审
    4. 决策 + 复盘
```

### Learnings

```
1. 反思 - 冲突早处理:
   2 周才介入太晚了
   1 周内就该出手
   leader 的敏感度要高

2. 反思 - 决策要客观:
   外部架构师 + 客观打分
   避免我个人偏见

3. 反思 - 输的人也要照顾:
   给 "败方" 主导落地的机会
   他不会觉得是输
   反而成长
```

### 面试用法

```
被问 "讲一次冲突处理"
→ 用这个案例
→ 重点: 客观 + 流程 + 关系
→ 量化: 2 周分歧 / 1 周解决 / 双方成长
```

## 六、案例 5：跨部门推日志规范

### Situation

```
背景:
  公司 5 个 BU
  每个 BU 日志格式不同
  → 统一 ELK 难
  → 排查跨 BU 问题 N 倍工作量
  → 故障复盘累
  
  我作为基础架构组 TL
  想推 "公司级日志规范"
  
  困难:
  - 5 个 BU 各自历史
  - 老板们都忙
  - 推不动
```

### Task

```
6 个月内:
  - 全公司 80% 服务接入新规范
  - 不影响业务迭代
  - 被 5 个 BU 接受
```

### Action

```
Step 1: 内部痛点收集 (1 个月)
  - 跟 5 个 BU 各聊 2-3 个 owner
  - 收集 50+ 痛点案例
  - 量化损失 (排查时间 / 故障 MTTR)

Step 2: 设计规范 (1 个月)
  - 借鉴业界 (Spring / Logback / OpenTelemetry)
  - 设计字段标准 + 输出格式
  - 不挑战业务自由 (灵活字段保留)

Step 3: 提供工具 (1 个月)
  - 写 SDK (Go / Java)
  - 提供 lint 工具
  - 提供迁移 demo

Step 4: 找盟友 (并行)
  - 找最痛的 BU 试点 (订单组)
  - 帮他们做 (人手 + 技术)
  - 1 个月内全部接入

Step 5: 数据 + 推广 (1 个月)
  - 订单组数据: 排查时间 -70% / 故障 MTTR -50%
  - 拉协调会 (其他 BU owner)
  - 老板们看到数据，态度转好

Step 6: 强力推 (2 个月)
  - 老板支持下 → 推到所有 BU
  - 提供 1 对 1 帮扶
  - 双周进度 review

Step 7: 治理 (持续)
  - lint 强制
  - 接入率监控
  - 不接入的服务故障复盘扣分
```

### Result

```
量化结果:
  - 6 个月 80% 服务接入
  - 1 年 95% 接入
  - 排查时间 -70%
  - 跨 BU 协作效率 +50%
  - 故障 MTTR -40%

业务价值:
  - 年省排查时间 = N 个工程师人月
  - 故障损失减少 X 万

我的收益:
  - 公司级影响力
  - 老板们认可
  - 拿到当年优秀
```

### Learnings

```
1. 反思 - 推动改革要找盟友:
   一个人推不动
   找最痛的人一起做
   做出数据 → 其他人自然跟

2. 反思 - 工具 + 文档 + 帮扶:
   光发规范没用
   要让人用得起来
   工具自动化 + 帮扶降低门槛

3. 反思 - 老板支持是杠杆:
   有数据 + 有试点 → 才能拿到老板支持
   有了老板支持 → 推得快 5x

4. 反思 - 持续治理:
   推完不管 → 半年后退化
   要有监控 + 强制 + 复盘
```

### 面试用法

```
被问 "跨部门推动 / 影响力"
→ 用这个案例
→ 重点: 找盟友 + 数据驱动 + 持续治理
→ 量化: 5 BU / 80% 接入 / 排查时间 -70%
```

## 七、案例 6：救一个濒临失败的项目

### Situation

```
背景:
  公司核心项目"会员体系重构"
  原 PM + 1 个 TL 带
  6 个月后:
  - 进度 30% (原计划 60%)
  - 上线时间推迟 3 次
  - 团队士气低落
  - 老板要砍项目

  老板找我:
  - "你能接手吗？"
  - "1 个月看能不能救"
```

### Task

```
1 个月内:
  - 评估能否救
  - 救：定计划 + 完成上线
  - 不能救：清晰说出原因 + 止损方案
```

### Action

```
T+0 第 1 周: 摸底
  - 跟原 PM / TL 1v1
  - 跟团队每人 1v1
  - 看代码 / 文档 / 进度
  
  发现:
  - 需求范围过大 (原想做 100 功能上 100)
  - 团队能力不匹配 (没做过这种规模)
  - 沟通混乱 (产品 / 开发对不齐)
  - 没有清晰里程碑

T+1 周: 评估
  - 能救但需要调整
  - 给老板汇报: 救的方案 + 风险

T+2 周: 重启
  
  砍范围:
  - 100 → 30 个核心功能 (P0)
  - 70 个 P1 / P2 后续做
  - 跟产品 PK 砍掉
  
  补人:
  - 申请 2 个 senior 加入
  - 老人 + 新人配对
  
  改流程:
  - 每周里程碑 (清晰可衡量)
  - 每日站会
  - 卡点 24h 内升级
  
  调心态:
  - 跟团队讲清"我们能做成"
  - 给信心
  - 立 quick win (第 1 周交付一个)

T+3-4 周: 看效果
  - 进度按计划
  - 团队士气回升
  - 老板看到希望

后续 2 个月: 完成上线
  - 30 个功能按时上
  - 0 重大故障
  - 团队认可我

T+3 月: 复盘
  - 分析为什么之前不行
  - 总结改进
  - 给老板备忘
```

### Result

```
项目:
  - 1 个月救活，2 个月按时上
  - 30 个核心功能 100% 完成
  - 0 P0 故障
  - 业务方满意度高

我:
  - 老板信任增强
  - 后续接更多项目
  - 团队认可

复盘:
  - 砍范围是关键 (不是堆人)
  - 沟通透明 / 流程清晰
  - 团队心态比能力更重要
```

### Learnings

```
1. 反思 - 砍范围需要勇气:
   100 个功能上 30 个
   产品 / 老板都有压力
   但是真理: 少而精 > 多而废

2. 反思 - 心态决定一半:
   团队没信心 → 再多人也救不了
   leader 必须给信心 + 清晰方向

3. 反思 - 我接手后做的事:
   - 先摸底（不要一上来就干预）
   - 再决策（救 / 不救）
   - 然后执行（坚决 + 清晰）
   
   摸底太重要，避免错误决策

4. 反思 - 救火型 leader:
   公司需要这种能力
   但平时也要做"防火"
   我后续在团队推 "项目健康度监控"
   提前发现问题
```

### 面试用法

```
被问 "讲一次救火 / 接手烂摊子"
→ 用这个案例
→ 重点: 摸底 + 砍范围 + 心态 + 流程
→ 量化: 30% → 100% / 1 个月救活
```

## 八、案例 7：1 个 junior 被我从普通带到核心

### Situation

```
背景:
  2023 年招了赵六 (3 年经验)
  入职时:
  - 技术中等
  - 不太自信
  - 做事被动 (等任务)
  - 跟产品不敢沟通
```

### Task

```
1 年内:
  - 让赵六成为团队核心
  - 独立 owner 一个模块
  - 能跨团队沟通
```

### Action

```
Mentor 计划:

第 1-3 月: 打基础
  - 配老 senior 做 mentor
  - 让做小任务 (代码 review 严格)
  - 每周 1v1 (技术 + 心态)
  - 推荐学习资料 (书 + 课程)

第 4-6 月: 独立任务
  - 给 owner 1 个小模块
  - 让他自己跟产品对齐 (我后面看)
  - 出问题时不直接帮，而是引导思考
  - 1v1 频率降到双周

第 7-9 月: 主导项目
  - 让他主导一个项目 (5 人)
  - 我做"教练"角色
  - 季度技术分享 (强制他出来讲)

第 10-12 月: 跨团队
  - 让他参加跨部门会议
  - 让他写技术文档对外
  - 推他做内部分享

激励:
  - 月度突出贡献奖
  - 季度晋升评审重点推
  - 涨薪 + 期权倾斜
```

### Result

```
1 年后赵六:
  - 从 P5 升 P6+
  - 独立 owner 订单核心模块
  - 主导一个项目 (5 人)
  - 跨部门小有名气
  - 团队公认"潜力新星"

业务价值:
  - 团队多一个核心
  - 减少我亲自带项目的时间
  - 赵六可以教其他 junior

我的收益:
  - 培养能力被认可
  - 老板觉得我"会用人"
  - 团队梯队化更稳
```

### Learnings

```
1. 反思 - 培养是长期投资:
   1 年才看到效果
   不能急于求成

2. 反思 - 给空间 + 给挑战:
   让他自己解决 (而不是帮他做)
   适度的挑战 (踮脚能够到)
   失败的 OK (在容错范围内)

3. 反思 - 心态比技术先:
   先建立自信
   再提升技术
   情绪稳定 → 技术增长更快

4. 反思 - 培养是杠杆:
   带 1 个核心 = 我节省 50% 时间
   带 5 个核心 = 整个团队 leverage
```

### 面试用法

```
被问 "怎么培养下属 / Mentoring"
→ 用这个案例
→ 重点: 阶段性 + 给空间 + 心态
→ 量化: 1 年从 P5 → P6+ / 独立 owner
```

## 九、案例 8：跟老板说不（但赢得尊重）

### Situation

```
背景:
  老板要求 1 周内上一个新功能
  我评估: 至少 3 周 (含测试 + 灰度)
  
  压力:
  - 老板态度强硬
  - 同事都不敢说"做不了"
  - 上面有大老板施压
```

### Task

```
- 把真实情况说清
- 不让团队加班赶
- 维护跟老板关系
- 找折中方案
```

### Action

```
T+0: 跟老板 1v1 (闭门 30 分钟)

我说:
  "老板这事我有想法，您听听:
   
   1. 真实情况:
      完整版 3 周 (开发 1.5 周 / 测试 0.5 周 / 灰度 1 周)
      1 周做完意味着跳过测试 / 灰度
      上线后大概率出故障
      
   2. 我的备选 3 个方案:
      A. 完整 3 周（推荐）
         风险低 / 团队不加班 / 体验好
      
      B. 1 周 MVP（70% 功能）
         跳过 P2 功能
         有测试 + 简化灰度
         可控风险
      
      C. 1 周完整版
         加班 + 跳过测试
         大概率故障
         我反对
   
   3. 我的建议:
      选 B 上 MVP，2 周后补全
      你看行吗？"

老板反应:
  - 一开始还坚持完整版 1 周
  - 我给数据 (类似项目失败案例)
  - 老板认了 B 方案

T+0 跟团队:
  - 同步老板决策
  - 1 周内做 MVP
  - 不加班
```

### Result

```
项目:
  - 1 周 MVP 上线
  - 0 故障
  - 用户反馈 OK
  - 2 周后完整版上

我:
  - 老板对我信任增强
  - 后续类似事情先问我
  - 团队佩服我"敢说话"

复盘:
  - "say no" 不是真说 no
  - 是给老板更好选项
  - 让老板做决策
```

### Learnings

```
1. 反思 - 不要只说"做不了":
   老板要的不是 yes 或 no
   而是数据 + 选项 + 推荐

2. 反思 - 私下 PK:
   不在公开场合反驳
   老板需要面子

3. 反思 - 给数据不给情绪:
   "做不了"是情绪
   "需要 3 周因为 X / Y / Z"是数据

4. 反思 - 让老板赢:
   老板拍板
   我执行
   出了问题我兜底
   老板觉得"这个 leader 靠谱"
```

### 面试用法

```
被问 "跟老板意见不一致 / 怎么 say no"
→ 用这个案例
→ 重点: 数据 + 选项 + 让老板拍板
→ 量化: 1 周 vs 3 周 / 0 故障
```

## 十、案例 9：处理 P0 故障 + 复盘

### Situation

```
背景:
  某周二下午 4 点 (业务高峰)
  监控告警: 订单接口错误率从 0.1% 飙到 30%
  钉钉群炸开
  老板拉群
```

### Task

```
- 紧急止血
- 找根因
- 完整复盘
- 长期改进
```

### Action

```
T+0 5 分钟: 战时指挥成立
  - 拉所有 owner (订单 / DBA / 运维 / 网络)
  - 开实时监控大盘
  - 评估影响 (5w 用户 / 100w GMV/分钟)

T+0 8 分钟: 一级降级
  - 关闭推荐 / 评论
  - 减少 DB 压力

T+0 10 分钟: 限流
  - 网关限到 5k QPS

T+0 15 分钟: 部分恢复

T+0 20 分钟: 定位
  - SHOW PROCESSLIST 看 DB 慢查询
  - 找到根因: 营销新接口全表扫

T+0 30 分钟: 根因修复
  - KILL 慢查询
  - 通知营销下线该接口
  - DB CPU 恢复

T+0 45 分钟: 完全恢复

T+1 day: 复盘文档

T+1 week: 改进方案
  - SQL Review 流程化
  - 慢查询自动告警
  - 限流自动触发
  - 团队培训

T+1 month: 改进落地
T+3 month: 类似事故 0 复发
```

### Result

```
本次:
  - 影响: 5w 用户 / 100w GMV
  - 恢复时间: 45 分钟
  - 没有 escalate 到大老板

后续:
  - SQL Review 拦截 200+ 高危 SQL
  - 类似事故 0 复发
  - MTTR 从 1h → 15min
  - 团队应急能力提升

我的成长:
  - 战时指挥能力被认可
  - 老板把更多关键时刻交给我
  - 公司级 SRE 培训邀请我讲
```

### Learnings

```
1. 反思 - 止血 > 定位:
   先恢复服务 (15 分钟内)
   再找根因
   不要为定位耽误恢复

2. 反思 - 战时指挥不是 1 个人:
   多 owner 协作
   各自负责
   总指挥统筹

3. 反思 - 复盘比故障重要:
   故障是付学费
   复盘是吸收教训
   1 个故障 → 5 个改进 → 防止 N 个故障

4. 反思 - 长期改进比短期修复重要:
   SQL Review 流程
   慢查询告警
   团队能力建设
   这些才是长期稳定
```

### 面试用法

```
被问 "讲一次故障 / 救火"
→ 用这个案例
→ 重点: 止血 + 流程 + 复盘
→ 量化: 45 分钟恢复 / 改进后 0 复发
```

## 十一、案例 10：推动 Code Review 文化

### Situation

```
背景:
  团队 12 人
  Code Review 形同虚设:
  - PR 几小时合 (没人认真看)
  - 大家觉得"浪费时间"
  - 故障频发 (本可 review 拦截)
  - 我作为 TL 烦恼
```

### Task

```
3 个月内:
  - 团队真正做 Code Review
  - 提升代码质量
  - 减少线上故障
```

### Action

```
Step 1: 找痛点
  - 拉了最近 3 个月故障数据
  - 80% 故障可在 review 拦截
  - 每次故障损失 X 小时
  - 团队会上分享数据
  - 大家共识: 必须做

Step 2: 标准化 Review
  - 我写 Review checklist (10 点)
  - 培训 1 次 (团队全员)
  - PR 模板 (描述 / 测试 / 影响)

Step 3: 制度化
  - 必须 2 人 approve
  - 24h 内响应
  - Junior 由 mentor review
  - Senior 互 review

Step 4: 工具化
  - lint 自动跑 (CI 拦截基础问题)
  - SonarQube 检测复杂度
  - 覆盖率门槛 (新代码 80%)

Step 5: 文化化
  - 鼓励质疑 (不针对人)
  - 分享心得 (Review 是学习)
  - 月度优秀 reviewer 评选
  - 我亲自 review 关键 PR

Step 6: 数据反馈
  - PR 平均 review 时长
  - 评审拦截 bug 数
  - 故障率变化
  - 月度 share 进展
```

### Result

```
3 个月后:
  - PR 平均 review 时间: 5min → 2h (认真看了)
  - PR 平均 comment 数: 0 → 8
  - Review 拦截 bug 数: 月 50+
  - 线上故障: 月 5 → 月 1
  - 团队满意度提升

6 个月后:
  - Review 文化稳固
  - 新人上手快 (review 中学习)
  - 代码质量明显提升
  - 推到全公司

我的收益:
  - 团队认可
  - 公司级文化推动者
  - 老板看到长期价值
```

### Learnings

```
1. 反思 - 文化不是喊口号:
   要数据 + 工具 + 制度 + 激励
   单方面强推没用

2. 反思 - Leader 要带头:
   我自己每天花 1h Review
   其他人才会重视
   不能 leader 自己不做

3. 反思 - 不针对人:
   Review 是 review 代码
   不是攻击人
   团队氛围才好

4. 反思 - 长期主义:
   3 个月才见效
   不要急
   长期受益巨大
```

### 面试用法

```
被问 "怎么推团队改进 / 推 Code Review"
→ 用这个案例
→ 重点: 数据 + 制度 + 工具 + 带头
→ 量化: 故障 5 → 1 / 拦截 50+ bug
```

## 十二、案例 11-15：快速版

### 案例 11：拒绝晋升一个不合格的下属

```
背景: 张三技术好但合作差，自己想晋升 P6
我做的:
  - 数据收集 (协作反馈 / 项目结果)
  - 1v1 直接讲 (不合格的 3 点)
  - 给改进方向 + 时间
  - 不通过 (不是因为讨厌他，是真不合格)
结果: 张三半年改进显著，1 年后真正合格晋升
反思: 公平比讨好重要。短期得罪人 = 长期赢得尊重
```

### 案例 12：跟产品共创"对账系统"

```
背景: 财务每天对账 4h，痛点
我做的:
  - 主动找财务总监聊
  - 一起设计方案 (财务知需求 / 我懂技术)
  - 立项 → 上线
  - 每周联调
结果: 对账时间 4h → 30min / 资损减少 99%
反思: 技术 leader 不能只在工程师圈子
   主动接触业务方收益巨大
```

### 案例 13：跨地域团队管理

```
背景: 我团队北京 + 上海 + 新加坡 (10 人)
我做的:
  - 异步沟通为主 (Notion / Slack)
  - 每周 1 次全员 (1h 重叠)
  - 季度 offsite 团建
  - 重要决策必文档化
结果: 团队默契 / 0 关键人离职 / 业务推进顺
反思: 跨地域 = 流程比天赋重要 / 文化对齐先于工作
```

### 案例 14：把团队从 Java 转 Go

```
背景: 老 Java 系统，团队 Java 老员工
我做的:
  - 数据驱动 (Java 性能瓶颈)
  - POC 1 个月 (Go 验证)
  - 老员工 mentor (慢慢学)
  - 试点 → 推广
  - 6 个月完成迁移
结果: 性能 5x / 成本 -60% / 团队 Go 能力建立
反思: 技术栈迁移 = 业务驱动 + 团队耐心 + 数据说话
```

### 案例 15：跨部门救一个核心服务

```
背景: 兄弟团队的服务挂了 4 小时，他们解决不了
我做的:
  - 主动 offer 帮忙 (他们 owner 不在)
  - 我团队抽 2 人协助
  - 1 小时定位 + 修复
  - 不抢功劳
结果: 兄弟团队感激 / 老板看到我"主动" / 跨部门关系建立
反思: 互助文化 = 长期跨部门资本
   不要看小团队 silo / 公司是一家
```

## 十三、面试加分点

- **15+ 真实案例** 不只是方法论
- **每个案例 STAR + 反思** 完整结构
- **量化数据**多维度（团队 / 项目 / 业务）
- **失败 / 反思真实可信**
- **流程化 + 文化化** 体系思考
- **跨部门 / 跨地域** 复杂场景
- **辞退 / 拒绝 / say no** 敢做难决策
- **救火 / 救项目 / 救人** 危机处理
- **培养 + Mentoring** 长期投资
- **量化下属成长**（晋升 / 留存 / 影响力）
